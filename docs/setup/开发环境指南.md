## 一、写代码前必须对齐的 10 件事

| # | 事项 | 谁来定 | 建议 | 状态 |
|---|------|--------|------|------|

| 5 | **LLM 选型** | 一起定 | 被测 Agent 用什么模型？裁判 LLM 用什么？变异 LLM 用什么？ | ❓ |
| 6 | **通信方式** | 一起定 | 开发阶段：Python 直接 import；验收阶段：REST API | ❓ |
| 7 | **时间线** | 一起定 | 各模块内部开发 deadline → 联调窗口 → 验收日期 | ❓ |

## 三、Python 版本与运行环境

### 3.1 版本

```
Python 3.11.x（建议 3.11.9）
```

**为什么是 3.11 而不是 3.12？**
- 3.12 发布不到两年，`captum`（模块三用）、部分 `transformers` 子包可能未适配
- 3.11 有 `X | None` 语法、`tomllib` 标准库、更好的错误提示
- 3.9/3.10 需要写 `Optional[X]` 和 `Union[X, Y]`，语法噪音大

### 3.2 虚拟环境

```bash

# 激活
source .venv/bin/activate         # Linux/Mac
```

模块二的沙箱测试依赖 Docker。其他两个模块不需要。

```bash

# 如果报 "command not found" 或 "daemon not running"：
#   Mac:      安装 Docker Desktop 或 colima
```

### 3.4 Python 语法约定

| | 废弃写法（不要用） | 统一写法 |
|---|-------------------|---------|
| 可选类型 | `Optional[str]`, `Union[str, None]` | `str \| None` |
| 联合类型 | `Union[int, str]` | `int \| str` |
| 列表类型 | `List[str]`（从 typing import） | `list[str]` |
| Future annotations | `from __future__ import annotations` | 不加（3.11 原生支持） |
| 读取 toml | `import tomli` | `import tomllib`（标准库） |
| 类型守卫 | `assert isinstance(x, str)` | 推荐 match/case |

### 4.2 Lock 文件

```bash
# 生成 lock 文件（由一个人统一做，其他人只消费）
pip-compile pyproject.toml -o requirements.lock
pip-compile pyproject.toml --extra dev -o requirements-dev.lock

# 安装（所有人都这样装）
pip-sync requirements.lock requirements-dev.lock
```

**规则**：
- CI 和联调环境**必须**用 lock 文件安装，保证三人版本完全一致
- lock 文件变更单独 PR，三个人都要 review
- 每周更新一次（周五下午），由指定的人做

### 4.3 新增依赖的流程

任何人想加新库，走这个流程：

```
1. 在自己的 pyproject.toml 的 [project.optional-dependencies] 里加项
   → 如果是三个模块都用的，加在 [project.dependencies]
   → 如果是某模块专用的，加在 [project.optional-dependencies.wp{N}]

2. 在分支上重生成 lock 文件：
   pip-compile pyproject.toml -o requirements.lock
   pip-compile pyproject.toml --extra dev -o requirements-dev.lock

3. 提 PR，标题加 [deps] 前缀，描述为什么加这个库

4. 至少一个其他人 review 并 approve

5. 合入后，所有人拉最新代码并重新 pip-sync
```

**禁止事项**：
- ❌ 不经过 lock 文件直接在本地 `pip install xxx`
- ❌ 在自己的模块里覆盖共享依赖的版本
- ❌ 加不兼容 3.11 的库

### 5.5 Schema 变更约定（`shared/schema/`）

`shared/schema/` 中的 Pydantic 模型是**三个模块的接口契约**。变更必须遵守以下规则：

**兼容性规则：**

| 变更类型 | 是否允许 | 条件 |
|---------|---------|------|
| 新增字段 | ✅ 允许 | 必须有默认值。所有下游模块不传新字段也不会挂 |
| 删除字段 | ❌ 禁止 | 除非三个人都确认没人用它 |
| 重命名字段 | ❌ 禁止 | 先新增后废弃，两版本后再删旧的 |
| 修改字段类型 | ❌ 禁止 | 比如 `str` 改 `int`。必须新增一个字段，旧字段标记废弃 |
| 修改枚举值 | ⚠️ 谨慎 | `str, Enum` 新增值是 OK 的。删除/重命名旧值是 breaking |
| 修改嵌套模型的字段 | 同上规则 | 递归应用 |

**变更流程：**

```
1. 在 shared/schema/ 中修改 Pydantic 模型
2. 更新 shared/__init__.py 中 __version__ = "0.1.0" → "0.2.0"
   （minor 版本号 +1 表示 schema 有新增字段；major +1 表示 breaking change）
3. 跑全部三个模块的测试，确保没有下游模块被 break
4. 提 PR，标题加 [schema] 前缀，三个人都 review
5. 合入后通知所有人拉最新代码
```

**Schema 版本号规则：**

```python
# shared/__init__.py
__version__ = "0.1.0"    # MAJOR.MINOR.PATCH
# MAJOR: breaking change（删字段、改类型）
# MINOR: 新增字段、新增模型
# PATCH: docstring 修正、validator 调整（不影响外部）
```

## 六、配置体系

### 6.1 配置层级和密钥安全

```
config/default.yaml          ← 默认值（git 跟踪，不含密钥）
       ↓ 继承并覆盖
config/staging.yaml          ← 联调环境（git 跟踪，不含密钥）
       ↓ 继承并覆盖
config/local.yaml            ← 本地开发（.gitignore，含密钥）
       ↓ 最高优先级
环境变量（TRACE_*）           ← 最高优先级，覆盖以上所有
```

**密钥安全红线**：
- ❌ 任何 `.yaml` 文件（含 staging）**绝不**存放真实 API key
- ✅ 密钥一律走 `config/local.yaml`（gitignored）或环境变量
- ✅ staging.yaml 只含非敏感配置（日志级别、开关、超时值）
- ✅ `config/local.yaml.example` 是模板（git 跟踪，值为 `"sk-xxx..."` 占位符）

```python
import yaml
from pathlib import Path

def _get_project_root() -> Path:
    """定位项目根目录。

    优先级：
      1. 环境变量 TRACE_PROJECT_ROOT（CI / 部署环境）
      2. 从 shared/utils.py 的位置向上找到包含 config/ 的目录
    """
    env_root = os.getenv("TRACE_PROJECT_ROOT")
    if env_root:
        return Path(env_root)

    # 从当前文件向上查找
    current = Path(__file__).resolve().parent
    for _ in range(5):
        if (current / "config" / "default.yaml").exists():
            return current
        current = current.parent

    raise RuntimeError(
        "Cannot locate project root. "
        "Set TRACE_PROJECT_ROOT env var, or ensure config/default.yaml exists."
    )

def load_config(env: str | None = None) -> dict:
    """
    加载配置，按 default → {env} → local → 环境变量 逐层覆盖。
    """
    config_dir = _get_project_root() / "config"

    # 1. 加载 default
    with open(config_dir / "default.yaml") as f:
        config = yaml.safe_load(f) or {}

    # 2. 覆盖环境配置
    env = env or os.getenv("TRACE_ENV", "local")
    env_path = config_dir / f"{env}.yaml"
    if env_path.exists():
        with open(env_path) as f:
            _deep_merge(config, yaml.safe_load(f) or {})

    # 3. 覆盖本地配置
    local_path = config_dir / "local.yaml"
    if local_path.exists():
        with open(local_path) as f:
            _deep_merge(config, yaml.safe_load(f) or {})

    # 4. 环境变量覆盖（最高优先级）
    _apply_env_overrides(config)

    return config

def _deep_merge(base: dict, override: dict) -> None:
    """递归合并 override 到 base，原地修改。"""
    for key, value in override.items():
        if key in base and isinstance(base[key], dict) and isinstance(value, dict):
            _deep_merge(base[key], value)
        else:
            base[key] = value

def _apply_env_overrides(config: dict) -> None:
    """环境变量 TRACE_* 覆盖配置中的对应键。

    映射规则：TRACE_LOGGING__LEVEL → config["logging"]["level"]
    双下划线 __ 表示嵌套层级。
    """
    for env_key, env_val in os.environ.items():
        if not env_key.startswith("TRACE_"):
            continue
        # TRACE_LOGGING__LEVEL → ["logging", "level"]
        parts = env_key.removeprefix("TRACE_").lower().split("__")
        if len(parts) < 2:
            continue

        # 向下钻取
        node = config
        for part in parts[:-1]:
            if part not in node:
                node[part] = {}
            node = node[part]
        node[parts[-1]] = env_val
```

### 9.2 `shared/errors.py`

```python
"""
统一错误码体系。格式：{MODULE}-{CATEGORY}{NNN}
"""

from enum import Enum

class ErrorCode(str, Enum):
    # ── 通用错误（SHARED） ──
    INVALID_SCHEMA = "SH-V001"
    CONFIG_ERROR = "SH-V002"
    TIMEOUT = "SH-T001"

    # ── 模块一 ──
    KB_QUERY_FAILED = "WP1-K001"
    DSL_PARSE_ERROR = "WP1-D001"
    POLICY_NOT_FOUND = "WP1-P001"
    OPERATOR_COMPILE_ERROR = "WP1-O001"

    # ── 模块二 ──
    SANDBOX_ERROR = "WP2-S001"
    JUDGE_ERROR = "WP2-J001"
    MUTATION_FAILED = "WP2-M001"
    REPLAY_MISMATCH = "WP2-R001"
    COVERAGE_STALE = "WP2-C001"

    # ── 模块三 ──
    GRADIENT_UNAVAILABLE = "WP3-G001"
    CONCEPT_UNSTABLE = "WP3-C001"
    COUNTERFACTUAL_LOW_CONFIDENCE = "WP3-F001"


class TraceError(Exception):
    """TRACE-G 统一异常基类。"""

    def __init__(self, code: ErrorCode, message: str, detail: dict | None = None):
        self.code = code
        self.message = message
        self.detail = detail or {}
        super().__init__(f"[{code.value}] {message}")

    def to_dict(self) -> dict:
        return {
            "error": {
                "code": self.code.value,
                "message": self.message,
                "detail": self.detail,
            }
        }
```

### 9.3 `shared/utils.py` 补充：项目初始化

```python
# 加到 shared/utils.py 中
# 替代原来 constants.py 里的 mkdir 副作用

from pathlib import Path
from shared.constants import DATA_DIR_NAME, TRAJECTORY_DIR_NAME, REPORT_DIR_NAME, GOLDEN_SET_DIR_NAME

def init_project(root: Path | None = None) -> Path:
    """初始化项目目录结构。各模块在启动时调用一次。"""
    if root is None:
        root = _get_project_root()

    for subdir in [DATA_DIR_NAME, TRAJECTORY_DIR_NAME, REPORT_DIR_NAME, GOLDEN_SET_DIR_NAME]:
        (root / subdir).mkdir(parents=True, exist_ok=True)

    return root
```

## 十二、常见故障排查

### 12.1 `pip-sync` 报版本冲突

```
ERROR: Cannot install xxx because these package versions have conflicting dependencies.
```

**原因**：lock 文件过期或两个依赖互相冲突。

**解决**：
```bash
# 1. 拉最新代码
git pull

# 2. 重生成 lock 文件
pip-compile pyproject.toml -o requirements.lock
pip-compile pyproject.toml --extra dev -o requirements-dev.lock

# 3. 重新同步
pip-sync requirements.lock requirements-dev.lock

# 4. 如果还不行，重建虚拟环境
rm -rf .venv
python3.11 -m venv .venv
source .venv/bin/activate
pip install pip-tools
pip-sync requirements.lock requirements-dev.lock
```

### 12.2 `import shared.xxx` 找不到模块

```
ModuleNotFoundError: No module named 'shared'
```

**原因**：没装 `-e`（editable）模式，或 `pyproject.toml` 的 `[tool.setuptools.packages.find]` 没配好。

**解决**：
```bash
# 确保以可编辑模式安装
pip install -e ".[wp2,dev]"

# 验证包是否被识别
pip show trace-redteam
# 看输出中 Location 是否指向你的源码目录
```

### 12.3 Docker 沙箱报错

```
docker.errors.DockerException: Error while fetching server API version
```

**原因**：Docker daemon 没启动。

**解决**：
```bash
# 检查 Docker 是否运行
docker ps

# Mac:
#   打开 Docker Desktop 等待鲸鱼图标变绿
#   或用 colima: colima start

# Windows:
#   打开 Docker Desktop 等待右下角图标稳定

# Linux:
sudo systemctl start docker

# 如果还不行 — 开发时用 skip_sandbox 模式绕过
# 在 config/local.yaml 中设置:
# dev:
#   skip_sandbox: true
```

### 12.4 pre-commit 钩子不触发

**原因**：没执行 `pre-commit install`。

**解决**：
```bash
pre-commit install
# 验证钩子已安装
ls -la .git/hooks/pre-commit
# 手动触发一次确认
pre-commit run --all-files
```

### 12.5 `ruff format` 和同事的格式打架

**原因**：有人没用 pre-commit，或 IDE 的 formatter 配置和 ruff 不一致。

**解决**：
```bash
# 卸载冲突的 IDE formatter 插件（如 autopep8、black、isort 等）
# 在 VSCode 中设置：
#   "editor.formatOnSave": true,
#   "editor.defaultFormatter": "charliermarsh.ruff"
#   "[python]": {
#     "editor.defaultFormatter": "charliermarsh.ruff"
#   }

# 全量格式化一次，提交干净的基准
ruff format .
git add -A && git commit -m "[chore] ruff format baseline"
```

### 12.6 `local.yaml` 被 git 追踪了

```
# git status 显示 config/local.yaml 是 tracked 或 staged
```

**解决**：
```bash
# 从 git 跟踪中移除（但保留本地文件）
git rm --cached config/local.yaml
git commit -m "[chore] remove local.yaml from tracking"

# 确认 .gitignore 里有 config/local.yaml 这一行
grep "local.yaml" .gitignore
```

### 12.7 `mypy shared/` 报 strict 类型错误

**原因**：在 `shared/` 中写了不完整的类型标注。

**解决**：`shared/` 包强制 strict。看到 mypy 报错就按它的提示补类型标注。常见遗漏：
- 函数返回值没写 `-> None`
- `dict` 没写成 `dict[str, Any]`
- 可选参数没用 `str | None`

### 12.8 Windows 路径分隔符问题

`Path(__file__).parent.parent / "config"` 在 Windows 上用反斜杠构造路径，但 `yaml.safe_load` 和 `open()` 都能正确处理。**唯一需要注意的**：不要手动用 `"\\"` 或 `"/"` 拼路径，全部用 `pathlib.Path`。
