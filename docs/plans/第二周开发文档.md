## 1. 前置条件与第一周基线

第二周直接复用第一周已经验收的能力，不重新设计容器调度、通信协议和受控工具：

- 标准镜像 `trace-redteam-agent:week1` 及其实际镜像摘要记录能力。
- Python Docker SDK 一次性容器调度器。
- `network_mode="none"`、非 root、只读根文件系统和受限 tmpfs。
- Docker Exec 固定 RPC helper + 容器回环 HTTP JSON-RPC 2.0。
- `RedTeamExecutionEngine`、`RuntimeClient` 和 `LangGraphAdapter`。
- 虚拟文件系统、Fake Shell、Mock API 和工具策略层。
- 容器外 JSONL 轨迹、序号连续性校验和原子提交。
- 规则评分器和证据事件定位。
- 正常、失败、取消、超时、Runtime 崩溃和评分异常后的容器清理。

第一周验收证据：

- 非 Docker 单元/集成测试 34 项通过。
- 真实 Docker 测试 15 项通过。
- 宿主机与 Runtime 均使用 Python 3.11.9 完成验证。
- 缓存镜像下 `scheduler.create()`：P50 约 0.354 秒，P95 约 0.411 秒。
- 创建开始至 Runtime 健康就绪：P50 约 2.607 秒，P95 约 2.663 秒。
- 验收后无遗留沙箱容器。

第二周不得破坏上述隔离基线和第一周协议兼容性。

为避免第二周实现反向破坏第一周，新增能力必须遵守以下兼容约束：

- 第一周 `TraceEvent` 的 `execution_id`、`sequence`、`timestamp`、`event_type`、`source` 和 `data` 字段保持原义；第二周字段只能以可选字段形式追加。
- 第一周 `ExecutionRequest` 已有字段不改名、不改变必填性；录制参数通过新的可选 `recording` 对象传入。
- `X-Protocol-Version` 继续使用 `1`；新增 JSON-RPC 方法和可选参数不得影响第一周客户端调用已有方法。
- `DockerSandboxScheduler.create(execution_id, image_ref, limits)`、`wait_until_ready()` 和 `destroy()` 的公开签名保持不变。
- 容器外 `TrajectoryStore` 继续负责第一周 JSONL 轨迹的增量写入、序号连续性校验和原子提交；第二周通过组合或扩展方式增加产物索引，不另建一套互不兼容的轨迹落盘逻辑。
- 第二周模型必须能够读取第一周 `schema_version="1.0"` 的轨迹；旧轨迹缺少重放产物时应返回“不可 strict 重放”，而不是解析失败或补造状态。
- 第一周原有 49 项测试属于强制回归基线；后续若测试数量变化，应以“第一周基线测试套件全部通过”为验收口径，49 仅作为当前数量记录。

## 2. 本周目标

本周需要完成：

1. 为轨迹事件增加逻辑时间、输入摘要、输出摘要、状态摘要和检查点引用。
2. 定义确定性规范化规则和 canonical JSON 序列化规则。
3. 录制 Agent 初始状态、模型决策、工具调用、工具结果和虚拟环境状态。
4. 为虚拟文件系统、Fake Shell、Mock API 和 LangGraph 状态实现显式导出/恢复接口。
5. 生成版本化 `ReplayManifest`，通过摘要定位全部重放产物。
6. 实现 `strict` 重放：不重新调用非确定性模型，注入已录制的结构化模型决策。
7. 实现 `live` 重放：固定环境和工具响应，但允许重新调用模型或 Fake 模型。
8. 实现工具的 `execute_and_verify` 与 `stub_response` 两种重放策略。
9. 对 strict 重放的规范化行为轨迹、对齐检查点状态和最终状态进行比较，并输出首个分歧位置。
10. 在协议定义的稳定检查点建立断点。
11. 从检查点注入新 Prompt 或受控工具结果，产生具有父子关系的新分支轨迹。
12. 对缺少镜像、Manifest、检查点、模型决策或工具响应的情况明确失败。
13. 保证录制、重放、分支、失败和取消后仍无遗留容器。

其中，第二周核心验收路径为 `strict` 重放与 Prompt 分支；`live` 重放和模型决策/工具结果替换必须建立在核心路径稳定之后，不得以牺牲 strict 完整性为代价。

本周不实现：

- Docker CRIU 或虚拟机级进程内存快照。
- 任意机器指令级断点。
- 对未显式序列化的第三方库内部状态进行恢复。
- 覆盖率计算和覆盖率引导 Fuzzing。
- 语义变异器。
- LLM-as-Judge、黄金集和漂移检测。
- 对真实公网、真实生产 API 或真实敏感文件的录制与重放。

## 3. 第二周完成边界

第二周所称“确定性重放”必须限定在系统可控制的边界内：

```text
固定镜像与 Agent 版本
  + 固定初始虚拟环境
  + 固定 Prompt、seed、区域设置和逻辑时钟
  + 录制并注入模型结构化决策
  + 录制并重放/校验工具响应
  + 只在显式稳定检查点恢复状态
  = 声明边界内的确定性重放
```

以下状态属于第二周可恢复范围：

- 用户 Prompt、系统提示词版本和 Agent 版本。
- LangGraph 已声明状态字段。
- 当前节点、步数、待执行结构化 action 和最终回答。
- 虚拟文件系统中的文件、目录和内容摘要。
- Mock API 已消费响应和剩余响应队列。
- Fake Shell 的逻辑工作目录和模拟环境状态。
- 模型结构化决策和对应输入摘要。
- 工具调用参数、结果、策略判定和副作用摘要。
- 随机种子、可序列化 PRNG 当前状态或消费计数、时区、区域设置和逻辑时间。

以下状态不属于可恢复范围：

- Python 进程内存地址和对象引用。
- 打开的真实文件描述符。
- 真实网络连接和 Socket 状态。
- Docker daemon、宿主机或第三方服务内部状态。
- 未经过 Adapter 导出接口的 LangGraph/LangChain 私有对象。
- 任意线程调度、协程调度和操作系统时序。

验收口径必须使用：

> 在声明的确定性边界内，strict 重放产生的规范化行为事件序列摘要、对齐检查点状态摘要和最终受支持状态摘要一致。

不得笼统宣称整个大模型系统天然具备 100% 确定性。

## 4. 总体架构

### 4.1 录制流程

录制执行仍然复用第一周 `RedTeamExecutionEngine` 的调度和通信通道，通过 `ExecutionRequest.recording.enabled=True` 激活录制路径。关键差异在于**执行结束后、容器销毁前**，必须通过 Docker Archive API 将容器内生成的产物拉取到宿主机。

```text
TestCase / Prompt
  → RedTeamExecutionEngine
  → DockerSandboxScheduler 创建一次性容器
  → RuntimeClient 提交 recording_enabled=true 的 ExecutionRequest
  → 容器内 LangGraphAdapter（录制路径）
       ├─ DecisionRecorder 记录模型结构化决策
       ├─ ToolRecorder 记录调用、响应和副作用摘要
       ├─ StateCheckpointManager 在稳定边界导出状态到 /workspace/replay-out/
       └─ TraceCollector 输出带逻辑时间和摘要的事件
  → RuntimeClient 增量拉取轨迹事件（同第一周 execution.events）
  → TrajectoryStore 原子提交轨迹 JSONL（同第一周）
  → ArtifactTransfer.download(handle) 通过 get_archive() 拉取 /workspace/replay-out/
       ├─ 校验每个产物的 SHA-256 和 size_bytes
       └─ 写入内容寻址 ArtifactStore
  → ManifestBuilder 基于轨迹摘要和产物引用生成完整 ReplayManifest
  → 计算 manifest_digest 并写入 manifest.sha256（密封，此后不可变）
  → 调度器销毁容器
```

产物拉取必须在容器销毁前完成：容器使用 `network_mode="none"`，不存在其他通信通道。若 `get_archive()` 失败，Manifest 不得密封，运行标记为 `infrastructure_error`，容器仍须销毁。

### 4.2 strict 重放流程

```text
ReplayManifest
  → 校验 schema、摘要和产物完整性
  → 校验目标镜像摘要可用
  → 创建同镜像的一次性无网容器
  → 上传重放产物到容器 tmpfs
  → 恢复初始虚拟环境与 Agent 状态
  → RecordedDecisionModel 注入历史模型决策
  → ToolReplayer 执行并校验或直接桩回放
  → 生成新的行为轨迹和独立的重放审计事件
  → Normalizer 分别投影历史行为轨迹与重放行为轨迹
  → Comparator 比较行为摘要和受支持状态摘要并定位首个分歧
  → 保存 ReplayResult
  → 销毁容器
```

### 4.3 断点分支流程

```text
父 ReplayManifest + checkpoint_id
  → 校验检查点可恢复
  → 恢复检查点状态
  → 引用父轨迹稳定前缀并废弃父轨迹后缀
  → 注入新 Prompt / 模型决策 / 工具结果
  → Prompt 变异后的后缀默认以 live-and-record 模式重新执行
  → 仅在显式提供完整替代决策时允许 strict-with-replacements
  → 生成新 trajectory_id 和 replay_id
  → 记录 parent_trajectory_id、parent_replay_id、fork_sequence
  → 重新计算后续事件和状态摘要
```

分支轨迹的逻辑视图由”父轨迹 `0..fork_sequence` 的已校验前缀引用 + 注入事件 + 新执行后缀”组成。父轨迹前缀不需要再次伪装成子执行产生的原始事件，父轨迹后缀不得复制到子轨迹。`fork_sequence` 表示父轨迹中最后一个被继承的行为事件序号，注入发生在其后、下一稳定步骤执行前。

### 4.4 容器内 LangGraphAdapter 多模式改造

第一周 `LangGraphAdapter.execute()` 只有一条固定的执行路径。第二周需要在不破坏第一周路径的前提下，支持 5 条路径。通过在 `ExecutionRequest` 中新增的可选字段区分模式：

```python
class LangGraphAdapter(AgentAdapter):
    async def execute(self, request: ExecutionRequest) -> AsyncIterator[TraceEvent]:
        if request.recording and request.recording.enabled:
            return self._execute_recording(request)
        if request.replay:
            return self._execute_replay(request)
        # 第一周路径：无 recording、无 replay 时走原逻辑
        return self._execute_normal(request)
```

五种执行路径的数据来源对照：

| 路径 | 触发条件 | 模型来源 | 工具行为 | 状态来源 |
|---|---|---|---|---|
| 普通执行 | `recording is None` 且 `replay is None` | FakeChatModel | 正常执行 | 初始状态 |
| 录制执行 | `recording.enabled=True` | FakeChatModel + DecisionRecorder 包装 | 正常执行 + ToolRecorder 包装 | 初始状态 |
| strict 重放 | `replay.mode=”strict”` | RecordedDecisionModel（注入历史决策） | ToolReplayer（按交互策略执行并校验/桩回放） | Manifest 恢复的检查点状态 |
| live 重放 | `replay.mode=”live”` | FakeChatModel 重新调用 | ToolReplayer（固定响应，新请求标记分歧） | Manifest 恢复的检查点状态 |
| 分支 | `replay.fork` 参数存在 | FakeChatModel（新后缀重新调用） | 前缀回放 + 新后缀正常执行 | 指定检查点恢复 + 注入 |

`agent_image/app/replay/replay_adapter.py` 负责实现录制和重放路径的具体逻辑，作为 `LangGraphAdapter` 的委托对象，不替代 `LangGraphAdapter`。`LangGraphAdapter` 本身保持为框架适配的唯一入口，第一周行为不变。

DecisionRecorder 和 ToolRecorder 对 FakeChatModel 和 ToolRegistry 做**装饰器包装**，不修改 FakeChatModel.plan() 的现有签名。录制时：
- DecisionRecorder 拦截 `plan()` 返回的 `(action_dict, loop_bool)`，加上 `decision_index`、`input_digest`、`output_digest`、前后检查点 ID，生成 `RecordedModelDecision` 写入内存缓冲。
- ToolRecorder 拦截 `ToolRegistry.execute()` 返回的 `ToolResult`，加上 `interaction_index`、参数摘要、结果摘要、副作用摘要，生成 `RecordedToolInteraction` 写入内存缓冲。
- 执行结束后，两套缓冲由 StateCheckpointManager 一并序列化到 `/workspace/replay-out/`。

### 4.5 容器内 Runtime 状态机扩展

第一周 `RuntimeState` 管理单次执行：`PENDING → RUNNING → SUCCEEDED/FAILED/CANCELLED/TIMED_OUT`。第二周新增的方法和状态：

```text
                    ┌──────────────────────────────────────────┐
                    │             RuntimeState                 │
                    │  (一容器只绑定一个 execution_id)          │
                    └──────────────────────────────────────────┘
                                    │
            ┌───────────────────────┼───────────────────────┐
            ▼                       ▼                       ▼
    execution.submit         replay.submit            replay.fork
    (第一周 + recording)     (strict / live)         (分支)
            │                       │                       │
            ▼                       ▼                       ▼
    PENDING → RUNNING        PENDING → RUNNING        PENDING → RUNNING
      → 终态                    → 终态                    → 终态
```

关键约束：

- `replay.submit` 和 `replay.fork` 与 `execution.submit` 共享”一容器只绑定一个 execution_id”约束：`_lock` 被 `submit()` 获取后，其他调用返回 `-32004`（Runtime 正忙）。
- 重放/分支路径不从 `ExecutionRequest.prompt` 启动 Agent，而是从 Manifest 恢复的检查点状态启动。启动入口由 `resume_phase` 字段决定（§10）。
- 录制执行结束后，Runtime 保持终态，宿主机通过 `get_archive()` 拉取 `/workspace/replay-out/` 中的产物。第二周当前实现复用第一周的长生命周期 Uvicorn 进程：Runtime 不会在单次执行终态后自行退出，容器生命周期本身保证产物拉取窗口，因此 `TRACE_G_KEEP_ALIVE_SECONDS` 本周不强制实现。未来若引入终态自动退出或优雅关闭，再增加该配置；容器销毁前已落盘的产物不受影响。
- `server.py` 的 `_dispatch()` 追加 `replay.submit`、`replay.fork`、`replay.checkpoints` 三个新路由，保留第一周四个方法不变。

## 6. 核心数据模型

所有新增模型必须包含 `schema_version`；对第一周既有模型追加的字段必须提供默认值，禁止破坏第一周已有轨迹的读取能力。第二周全新模型的核心字段可以保持必填。

### 6.1 重放模式

```python
from enum import StrEnum


class ReplayMode(StrEnum):
    STRICT = "strict"
    LIVE = "live"


class ToolReplayMode(StrEnum):
    EXECUTE_AND_VERIFY = "execute_and_verify"
    STUB_RESPONSE = "stub_response"


class ForkSuffixMode(StrEnum):
    LIVE_AND_RECORD = "live_and_record"
    STRICT_WITH_REPLACEMENTS = "strict_with_replacements"
```

### 6.2 产物引用

```python
from typing import Annotated, Literal, TypeAlias, Union

from pydantic import BaseModel, Field


Digest = Annotated[str, Field(pattern=r"^sha256:[0-9a-f]{64}$")]
ImageIdentity = Annotated[
    str,
    Field(pattern=r"^(?:.+@)?sha256:[0-9a-f]{64}$"),
]
JsonValue: TypeAlias = Union[
    None,
    bool,
    int,
    float,
    str,
    list["JsonValue"],
    dict[str, "JsonValue"],
]


class ArtifactRef(BaseModel):
    schema_version: str = "1.0"
    media_type: str
    sha256: Digest
    size_bytes: int = Field(ge=0)
    relative_path: str
```

`relative_path` 只允许使用 `/` 分隔的仓库内相对路径，必须拒绝空段、`.`、`..`、盘符、UNC 路径和反斜杠。内容寻址产物的路径应由 `sha256` 推导，而不是直接信任调用方输入。完整性以 `sha256` 和 `size_bytes` 共同校验，禁止在 Manifest 中保存宿主机绝对路径。

```python
    manifest_digest: Digest | None = None

```

`ReplayManifest` 是不可变的录制包描述，不携带本次运行选择的 `strict/live` 模式。重放模式属于每次重放请求，避免 Manifest 中的 `mode` 与 CLI 参数发生冲突：

CLI/API 接收 `ForkInjection`，宿主机校验并计算摘要后转换为 `RecordedForkInjection` 写入子 Manifest 产物，不能信任调用方自行提交的摘要、操作者或时间。

`manifest_digest` 按 `model_dump(exclude={"manifest_digest"})` 的 canonical JSON 计算。验证时：读取 Manifest JSON → 删除 `manifest_digest` 键 → canonical JSON 序列化 → SHA-256 → 与 `manifest.sha256` 比较。`created_at` 是证据元数据，属于 Manifest 完整性保护范围，但不进入行为轨迹一致性摘要。

**Manifest 两阶段构建**：35+ 个字段的填充时间点不同，构建必须分阶段：

| 阶段 | 时机 | 可填充字段 |
|---|---|---|
| Pre-exec | 录制开始前 | `case_id`、`scenario_id`、`seed`、`image_ref`、`agent_version`、`graph_version`、`system_prompt_version`、`tool_registry_version`、`policy_version`、`state_codec_version`、`normalization_version`、`canonical_json_version`、`default_tool_replay_mode` |
| Post-create | 容器创建后 | `image_digest`、`image_digest_kind`、`runtime_version`、`protocol_version`（从 SandboxHandle 和 /health 获取） |
| Post-exec | 执行完成且产物拉取后 | `prompt_digest`、`initial_state_digest`、`normalized_behavior_trace_digest`、`determinism_config_digest`、`system_prompt_digest`、所有 `ArtifactRef` |
| Seal | 所有字段就绪后 | `manifest_digest`（计算后填入并写入 manifest.sha256） |

Post-exec 阶段的字段依赖产物已通过 ArtifactStore 校验，任何产物缺失或摘要不一致时 Manifest 不得进入 Seal 阶段。

**`initial_state_digest` 计算范围**：对录制开始时采集的 `CheckpointStateEnvelope`（§9.1）整体计算 canonical JSON SHA-256，包含 `agent_state`、`virtual_filesystem_state`、`fake_shell_state`、`mock_api_state`、`rng_states` 和 `environment` 白名单字段。这等价于执行开始前的全状态快照摘要。检查点级别的 `state_digest`（§6.4）使用相同方法对单个检查点的 `CheckpointStateEnvelope` 计算，两者算法一致、范围不同（初始 vs 增量）。

为兼容第一周 `DockerSandboxScheduler._image_digest()` 的实际行为，`image_digest` 原样接受 `repo@sha256:...` 或本地 Docker image ID `sha256:...`，`image_digest_kind` 分别记录 `repo_digest` 或 `image_id`。不得把可漂移镜像 tag 当作 `image_digest`。

> **与总体文档 §8.2 Manifest 示例的区别**：总体文档的 Manifest YAML 示例中包含 `mode: strict` 字段。第二周设计将 `mode` 从 Manifest（录制产物）移至 `ReplayRequest`（每次重放的运行时参数）。Manifest 是不可变的录制包描述，不携带每次运行的 `strict/live` 选择。总体文档的 YAML 示例将在总集成阶段更新。

## 7. 轨迹事件扩展

第一周 `TraceEvent` 保持兼容，第二周新增可选字段：

```python
class TraceEvent(BaseModel):
    schema_version: str = "1.1"
    execution_id: str
    sequence: int
    timestamp: datetime
    event_type: str
    source: str
    data: dict = Field(default_factory=dict)

    logical_time: int | None = None
    input_digest: str | None = None
    output_digest: str | None = None
    state_digest: str | None = None
    checkpoint_id: str | None = None
```

规则：

- `sequence` 是单次执行内的全序编号。
- `logical_time` 由 Runtime 按行为语义步骤单调递增，不使用真实墙钟时间驱动重放；重放控制/审计事件不得推进参与比较的行为逻辑时间。
- `timestamp` 仅用于诊断，不进入 strict 核心摘要。
- 模型、工具和检查点事件必须尽可能填写输入、输出和状态摘要。
- 第一周事件缺少新增字段时仍可读取，但若缺少 strict 所需产物，必须明确判定为不可 strict 重放。

第二周新增事件类型：

| 事件类型 | 含义 |
|---|---|
| `checkpoint_created` | 已导出稳定检查点 |
| `state_restored` | 已恢复初始状态或检查点 |
| `model_decision_recorded` | 已保存结构化模型决策 |
| `model_decision_replayed` | strict 模式已注入历史决策 |
| `tool_response_recorded` | 已保存工具结果 |
| `tool_response_replayed` | 已桩回放工具结果 |
| `tool_effect_verified` | 已执行工具并验证副作用摘要 |
| `replay_started` | 重放开始 |
| `replay_matched` | strict 重放的行为与状态摘要一致 |
| `replay_diverged` | 重放发生分歧 |
| `fork_started` | 分支执行开始 |
| `fork_injection_applied` | 分支注入已应用 |

事件分为两条逻辑流：

1. **行为事件流**：描述 Agent 节点、模型决策、工具调用、工具结果、安全策略和终止状态，用于 strict 摘要比较。
2. **重放审计流**：描述 Manifest 校验、状态恢复、录制/桩回放动作、分支注入和清理结果，用于诊断和证据留存，但不直接进入行为摘要。

第一周 `TrajectoryStore` 的行为 JSONL 保持不变：序号必须从 0 连续递增，最后一条仍必须是 `execution_*` 终态事件。第二周不得在其终态事件之后追加 `replay_matched`、`state_restored` 等控制事件。

因此默认采用独立 `ReplayAuditStore`：

```python
class ReplayAuditEvent(BaseModel):
    schema_version: str = "1.0"
    replay_run_id: str
    audit_sequence: int = Field(ge=0)
    timestamp: datetime
    event_type: str
    data: dict[str, JsonValue] = Field(default_factory=dict)
```

`model_decision_recorded/replayed` 和 `tool_response_recorded/replayed` 属于行为事件，写入本次执行的行为轨迹，并在规范化时映射为阶段无关名称；`checkpoint_created`、`state_restored`、`tool_effect_verified`、`replay_*` 和 `fork_*` 写入独立审计流。检查点状态是否一致通过 `CheckpointComparison` 单独验证，不依赖审计事件序号完全一致。

## 8. Canonical JSON 与摘要规则

### 8.1 Canonical JSON v1

Canonical JSON v1 采用 RFC 8785/JCS 的确定性编码思想，并增加本项目的输入约束。所有参与摘要计算的数据必须先规范化：

- UTF-8 编码，不带 BOM。
- 字符串先统一为 Unicode NFC，再按 JCS 规则排序对象键和编码字符串。
- 使用紧凑分隔符 `,` 和 `:`，不写无意义空白。
- 时间统一转为 UTC ISO 8601，并使用 `Z` 后缀。
- 摘要输入中的业务数值优先使用整数；非整数必须先转换为定义明确的十进制定点字符串。禁止 NaN、Infinity 和负零。
- 字典不得通过 `str(dict)` 参与摘要。

Canonical JSON 编码器只负责稳定编码，不负责判断字符串是否为宿主机路径。Prompt、模型参数或测试载荷可以合法包含 `/etc/passwd`、`C:\\...` 等文本。宿主机绝对路径、路径穿越和反斜杠限制只在 `ArtifactRef.relative_path`、Manifest 路径、tar 成员名等具有路径语义的字段上校验。

参考接口：

```python
def canonical_json_bytes(value: object) -> bytes: ...


def sha256_digest(value: object) -> str:
    return "sha256:" + hashlib.sha256(canonical_json_bytes(value)).hexdigest()
```

### 8.2 事件规范化 v1

strict 对比时排除：

- `execution_id`、`container_id`、`replay_id` 等每次运行不同的标识。
- 真实 `timestamp`。
- 宿主机临时目录。
- Docker 自动生成名称和日志前缀。
- 性能耗时字段。
- `ReplayAuditStore` 中的全部重放控制事件。

strict 对比时保留：

- 相对事件顺序和 `logical_time`。
- `event_type` 和稳定 `source`。
- 节点名称。
- 模型决策摘要。
- 工具名称、规范化参数和结果类别。
- 策略判定和风险类别。
- 状态摘要和检查点类型。
- 终止类型。

录制和重放阶段的事件名称需要投影为相同的语义事件：

| 原始/审计事件 | 行为比较事件 |
|---|---|
| `model_decision_recorded`、`model_decision_replayed` | `model_decision` |
| `tool_response_recorded`、`tool_response_replayed` | `tool_response` |
| `tool_effect_verified` | 不单独进入行为流；其结果并入对应 `tool_result` 的副作用摘要 |
| `replay_started`、`state_restored`、`replay_matched`、`replay_diverged` | 仅审计流 |
| `fork_started`、`fork_injection_applied` | 仅审计流；注入后的新 Prompt/决策通过后续行为事件体现 |

Comparator 必须基于规范化后的“行为事件索引”定位首个分歧，并同时报告源轨迹原始 `sequence` 与重放轨迹原始 `sequence`，不能假设两条原始事件流序号完全相同。

规范化后的单个事件不得包含原始密钥或未脱敏敏感内容。

脱敏不能在录制完成后静默改写将被 strict 重放消费的 Prompt、模型决策或工具结果，否则摘要一致但执行语义已经改变。第二周只允许使用合成凭据和 Mock 敏感数据；若输入中发现真实密钥，应在执行前拒绝或替换并记录处理结果。未来若需要保留真实敏感响应，必须引入独立加密产物存储和访问审计；在此之前，不得将因脱敏而无法恢复原始语义的轨迹标记为 strict 可重放。

## 10. 检查点设计

稳定检查点只允许创建在：

1. 模型调用前。
2. 模型结构化决策产生后。
3. 工具调用前。
4. 工具结果和副作用提交后。
5. LangGraph 节点状态提交后。

检查点创建顺序：

```text
暂停进入下一稳定步骤
  → 导出 Agent 状态
  → 导出受控工具状态
  → 导出模型/工具消费游标和已登记 PRNG 状态
  → canonical JSON 序列化
  → 计算 state_digest
  → 原子写入 ArtifactStore
  → 产生 checkpoint_created 事件
  → 继续执行
```

检查点恢复顺序：

```text
校验 ArtifactRef 摘要
  → 校验 schema_version 和 state codec version
  → 创建全新容器（新的 execution_id 和 SANDBOX_TOKEN）
  → 恢复虚拟环境
  → 恢复 Agent 白名单状态（将状态中的 execution_id 替换为当前容器的 execution_id）
  → 恢复模型/工具消费游标和 PRNG 状态
  → 设置逻辑时间
  → 从 resume_phase 对应入口继续并使用 resume_sequence 生成下一行为事件
```

恢复时 `StateCheckpoint.execution_id` 是录制时的旧 ID，必须替换为当前容器的 `EXECUTION_ID` 后再注入 Agent 状态。此替换不影响 `state_digest` 的计算——`state_digest` 在录制时已固定，恢复时先替换再注入，不做反向校验。

任何无法导出的状态都必须写入 `non_recoverable_reasons`，该检查点不得用于分支。

各检查点类型的恢复语义必须固定，禁止依靠调用栈推测：

| kind | 恢复后下一动作 |
|---|---|
| `BEFORE_MODEL` | 使用 `next_model_decision_index` 取得下一决策或在分支模式调用模型 |
| `AFTER_MODEL` | 不重复消费模型决策，从已恢复的结构化 action 继续 |
| `BEFORE_TOOL` | 使用 `next_tool_interaction_index` 处理下一工具调用 |
| `AFTER_TOOL` | 不重复执行工具，从已恢复的工具结果和副作用后状态继续 |
| `NODE_COMMIT` | 从已提交图状态确定的下一节点继续 |

## 11. 模型决策录制与 strict 重放

### 11.1 录制

**超时与产物拉取时间窗口**：录制执行的总超时由 `ExecutionRequest.timeout_seconds` 控制，但只约束 Agent 执行阶段。第二周当前 Runtime 沿用第一周长生命周期 Uvicorn，不在执行终态后自动退出；宿主机在调度器销毁容器前均可通过 `get_archive()` 拉取 `/workspace/replay-out/`。`TRACE_G_KEEP_ALIVE_SECONDS` 仅作为未来终态自动退出机制的预留配置，本周不强制实现。若 Agent 执行被 `execution.timed_out` 终止，已产生的部分产物仍应拉取并标记为不完整录制（`recording_audit` 中记录被截断的产物列表），Manifest 设置 `recoverable` 检查点为 False。

模型适配器需要将模型输出转换为框架无关结构：

```json
{
  "decision_type": "tool_call",
  "tool_name": "read_file",
  "arguments": {
    "path": "/etc/passwd"
  },
  "final_answer": null
}
```

录制内容必须包含：

- 规范化模型输入摘要。
- 结构化模型输出。
- 输出摘要。
- 模型调用前检查点 ID、可选的模型决策后检查点 ID 和单调递增的 `decision_index`。
- 模型和提示词版本。

### 11.2 RecordedDecisionModel

strict 模式不调用 Fake 模型或真实模型，而是使用：

```python
class RecordedDecisionModel:
    def next_decision(
        self,
        decision_index: int,
        input_digest: str,
    ) -> RecordedModelDecision:
        ...
```

行为要求：

- 当前输入摘要必须与录制摘要一致。
- 决策必须按照检查点恢复的 `next_model_decision_index` 顺序消费。
- 决策数量不足或多余都视为分歧。
- 不允许按“最相似 Prompt”选择历史决策。
- 分歧时产生 `replay_diverged`，不得静默回退到 live 模式。
- 分支的 `live_and_record` 后缀不使用 `RecordedDecisionModel`；它重新调用允许的模型/Fake 模型，并将新决策写入子 ReplayManifest。

## 12. 工具重放策略

### 12.1 execute_and_verify

适用于确定性的受控工具：

- `read_file`
- `write_file`
- Fake Shell 的允许命令
- 确定性 Mock API

流程：

```text
校验 interaction_index、工具名称和参数摘要
  → 执行历史工具调用
  → 获取实际结果
  → 计算规范化结果摘要
  → 比较历史 result_digest
  → 比较副作用前后 state_digest
```

结果不一致时标记 `replay_diverged`。

### 12.2 stub_response

适用于不希望重复执行或历史响应必须固定的工具：

- 非确定性 Mock 响应。
- 将来接入的外部服务代理。
- 不可安全重复的模拟副作用。

流程：

```text
校验工具名称和参数摘要
  → 不执行真实工具
  → 返回录制结果
  → 若存在 state_delta_artifact，校验前置状态后原子应用状态增量
  → 校验应用后的副作用摘要
```

若当前工具请求与历史参数摘要不匹配，不能返回历史响应。

同一轨迹允许不同交互选择不同策略。ToolReplayer 必须以 `RecordedToolInteraction.replay_mode` 为准，Manifest 的默认策略不能覆盖已经录制的逐交互策略。无副作用桩响应可以只返回录制结果；有副作用桩响应必须携带可验证状态增量，禁止仅伪造返回值而不恢复受控环境。

## 13. ArtifactStore 与内容寻址

推荐结构：

```text
data/
├── artifacts/
│   └── sha256/
│       └── ab/
│           └── cd/
│               └── abcdef...
└── replays/
    └── replay-0001/
        ├── manifest.json
        ├── manifest.sha256
        └── runs/
            ├── run-0001/
            │   ├── result.json
            │   ├── trajectory.jsonl
            │   └── replay-audit.jsonl
            └── run-0002/
                ├── result.json
                ├── trajectory.jsonl
                └── replay-audit.jsonl
```

写入规则：

1. 数据先写入同目录临时文件。
2. 计算 SHA-256。
3. 校验大小和摘要。
4. 原子重命名到内容寻址路径。
5. 已存在相同摘要时复用，不重复写入。
6. Manifest 对除 `manifest_digest` 外的 payload 计算摘要并写入 `manifest.sha256`。
7. Manifest 最后原子提交；提交后按不可变对象读取，更新必须创建新的 replay ID。

读取规则：

- 先校验相对路径没有 `..`。
- 校验文件大小。
- 重新计算 SHA-256。
- 摘要不一致立即失败。
- 不允许自动下载缺失产物。
- Manifest 自身摘要不一致时，在创建容器前失败。

### 13.1 容器产物传输

由于容器为 `network_mode="none"` 且不挂载宿主机目录，大型状态产物不通过 JSON-RPC Base64 传输。

使用 Docker Archive API：

- `put_archive()`：把已校验重放输入上传到 `/workspace/replay-in`。
- `get_archive()`：从 `/workspace/replay-out` 拉取录制产物。

宿主机客户端接口 `ArtifactTransfer`（`src/sandbox/client/artifact_transfer.py`）：

```python
from sandbox.scheduler.models import SandboxHandle
from sandbox.replay.models import ReplayManifest
from sandbox.replay.artifact_store import ArtifactStore


class ArtifactTransfer:
    """通过 Docker Archive API 在宿主机和容器间传输录制/重放产物。"""

    def __init__(self, docker_client, artifact_store: ArtifactStore) -> None: ...

    async def download(
        self,
        handle: SandboxHandle,
        *,
        max_total_bytes: int = 256 * 1024 * 1024,
        max_file_bytes: int = 64 * 1024 * 1024,
    ) -> dict[str, bytes]:
        """从 /workspace/replay-out 拉取产物，校验 tar 安全性后返回原始字节。
        调用方负责计算摘要并交给 ArtifactStore 保存。"""
        ...

    async def upload(
        self,
        handle: SandboxHandle,
        manifest: ReplayManifest,
        *,
        max_total_bytes: int = 256 * 1024 * 1024,
    ) -> None:
        """将 Manifest 引用的产物打包上传到 /workspace/replay-in。
        只上传 strict/live 重放所需的产物引用。"""
        ...
```

限制：

- 只允许上述两个固定目录作为 tar 根。
- tar 成员必须是相对路径。
- 拒绝绝对路径、`..`、盘符、UNC 路径、设备文件和符号链接。
- 拒绝大小超过 `max_file_bytes` 的单个成员。
- 拒绝总大小超过 `max_total_bytes` 的 tar 流。
- 上传前必须通过 `ArtifactStore` 重新计算并校验所有产物的 SHA-256。
- 下载后调用方必须对返回字节重新计算 SHA-256 后再写入 ArtifactStore。

## 14. JSON-RPC 协议扩展

保持 `X-Protocol-Version: 1`，新增方法属于向后兼容扩展。第一周已有方法的参数和响应不得改变；第二周新增对象必须是可选字段或新方法。

职责边界：宿主机 `ReplayEngine` 使用 `replay_id` 查询 ManifestStore；容器 Runtime 不访问宿主机数据目录，也不根据 `replay_id` 自行查找产物，只接受已经通过 Docker Archive API 上传到 `/workspace/replay-in` 的相对路径。

Runtime 的 `_dispatch()` 只能追加 `replay.submit`、`replay.fork` 等分支，必须保留第一周 `execution.submit/get/events/cancel` 的行为。`replay.submit` 和 `replay.fork` 必须像第一周 `execution.submit` 一样校验请求中的 `execution_id` 与容器环境变量 `EXECUTION_ID` 一致，并保持“一容器只绑定一个执行”的约束。

第一周客户端当前会把所有 JSON-RPC error 转成普通 `ProtocolError`。第二周应在不改变既有调用方捕获方式的前提下，为 `ProtocolError` 增加可选 `code` 和 `data`，或派生兼容的 `ReplayProtocolError`，从而让 ReplayEngine 能区分 `-32101..-32112`；原有第一周代码继续捕获其共同基类即可。

### 14.1 录制执行

### 14.6 Runtime 状态机扩展

`RuntimeState` 在第一周基础上增加对重放和分支方法的支持：

**`replay.submit` 处理流程**：

```text
校验 execution_id 匹配容器 EXECUTION_ID
  → 获取 _lock（已在 submit 中）
  → 读取 /workspace/replay-in/{manifest_relative_path}
  → 校验 Manifest schema 和 manifest_digest
  → 根据 ReplayRequest.mode 选择路径：
      strict: 加载 model_decisions + tool_records
              → 从 initial_state 或指定 checkpoint 恢复状态
              → 创建 RecordedDecisionModel
              → 创建 ToolReplayer
              → 启动 LangGraphAdapter（重放路径）
      live:   加载 tool_records
              → 从 initial_state 恢复状态
              → 保留 FakeChatModel
              → 创建 ToolReplayer（固定响应模式）
              → 启动 LangGraphAdapter（重放路径）
  → 执行完毕后产生 ExecutionResult
  → 终态
```

**`replay.fork` 处理流程**：

```text
校验 execution_id 匹配
  → 获取 _lock
  → 读取 Manifest + 校验
  → 定位 checkpoint_id 对应的 StateCheckpoint
  → 校验 checkpoint.recoverable=True
  → 校验 injection.type ∈ checkpoint.allowed_injection_types
  → 恢复检查点状态（替换 execution_id）
  → 应用 ForkInjection（prompt_replace/append/model_decision_replace/tool_result_replace）
  → 注入内容重新经过 schema 校验和工具安全策略
  → 根据 suffix_mode 启动执行：
      live_and_record: FakeChatModel + 录制新后缀
      strict_with_replacements: RecordedDecisionModel（完整替代决策）
  → 产生新 trajectory_id 和 replay_id
  → 记录 parent_trajectory_id、parent_replay_id、fork_sequence
  → 终态后父前缀引用写入子 Manifest
```

**`replay.checkpoints` 处理流程**：

```text
校验 execution_id 匹配
  → 读取 Manifest
  → 遍历 checkpoints ArtifactRef 指向的 JSONL
  → 过滤并返回所有 StateCheckpoint（包含 checkpoint_id、sequence、kind、resume_phase、state_digest、recoverable、allowed_injection_types）
```

三个新方法都通过 `_require_execution()` 校验 execution_id，共享第一周的 `_lock`、幂等和冲突检测逻辑。

## 15. ReplayEngine 生命周期

宿主机 `ReplayEngine` 是第二周新增的编排层，**包装**第一周的 `RedTeamExecutionEngine`，而非替代它。

**与 `RedTeamExecutionEngine` 的关系**：

- `RedTeamExecutionEngine` 继续负责第一周的普通执行闭环（`run_case()`），签名和行为不变。
- `ReplayEngine` 持有 `RedTeamExecutionEngine` 的依赖（`scheduler`、`runtime`、`case_source`），并新增 `manifest_store`、`artifact_store`、`artifact_transfer`、`comparator`。
- 录制执行通过 `RedTeamExecutionEngine` 完成：在 `ExecutionRequest` 上设置 `recording.enabled=True`，由引擎走完调度-执行-轨迹拉取-评分-清理的完整闭环，`ReplayEngine.record()` 在闭环的"产物拉取"阶段介入。
- 重放和分支由 `ReplayEngine` 直接管理：不经过 `RedTeamExecutionEngine.run_case()`。

```python
class ReplayEngine:
    def __init__(
        self,
        config: WeekOneConfig,
        scheduler: Scheduler,
        runtime: Runtime,
        case_source: TemplateCaseSource,
        manifest_store: ManifestStore,
        artifact_store: ArtifactStore,
        artifact_transfer: ArtifactTransfer,
        comparator: Comparator,
    ) -> None: ...

    async def record(self, case: TestCase) -> ReplayManifest: ...
    async def replay(
        self,
        source_replay_id: str,
        mode: ReplayMode,
    ) -> ReplayResult: ...
    async def fork(
        self,
        parent_replay_id: str,
        checkpoint_id: str,
        injection: ForkInjection,
        suffix_mode: ForkSuffixMode = ForkSuffixMode.LIVE_AND_RECORD,
    ) -> ReplayManifest: ...
```

### 15.1 record() 生命周期

```python
handle = None
removed = True
failure = None
try:
    # 1. 生成用例和请求（同第一周）
    case = self.case_source.generate(template_id, seed=seed)
    request = ExecutionRequest(
        ...,
        recording=RecordingOptions(enabled=True, ...),
    )

    # 2. 创建容器（同第一周调度器）
    handle = await self.scheduler.create(execution_id, image_ref, limits)
    removed = False
    request = request.model_copy(update={"image_digest": handle.image_digest})

    # 3. 等待就绪并提交录制执行（复用 RuntimeClient）
    await self.scheduler.wait_until_ready(handle)
    await self.runtime.submit(handle, request)

    # 4. 拉取轨迹（同第一周）
    store = TrajectoryStore(output_dir, execution_id)
    async for page in self.runtime.poll_and_stream_events(handle, request):
        store.append(page.events)
    result = await self.runtime.get_result(handle, execution_id)
    trajectory = store.commit(
        final_sequence=result.final_sequence,
        trace_count=result.trace_count,
    )

    # 5. 拉取录制产物（第二周新增）
    artifacts_raw = await self.artifact_transfer.download(handle)
    artifacts = self.artifact_store.ingest_all(artifacts_raw)

    # 6. 构建 Manifest（两阶段：初始 → 补充 → 密封）
    manifest = ManifestBuilder.create_initial(case, request)
    manifest = ManifestBuilder.populate_post_exec(
        manifest,
        handle=handle,
        trajectory=trajectory,
        artifacts=artifacts,
    )
    manifest = ManifestBuilder.seal(manifest)
    self.manifest_store.save(manifest)

except Exception as exc:
    failure = exc
    # 录制失败不产生 Manifest；产物保留为 .partial 供排障
finally:
    if handle is not None:
        try:
            await self.scheduler.destroy(handle)
            removed = True
        except Exception as cleanup_exc:
            removed = False
            failure = cleanup_exc

if failure:
    raise InfrastructureError("recording failed") from failure
return manifest
```

步骤 5（产物拉取）和步骤 6（Manifest 构建）是第二周新增的关键路径。它们必须在 `finally` 销毁容器**之前**完成（§4.1）。若步骤 5-6 失败，容器仍然销毁，但 Manifest 不生成。

### 15.2 strict/live 重放生命周期

```python
handle = None
trajectory = None
comparison = None
failure = None
removed = True
replay_run_id = id_factory.new_replay_run_id()
try:
    manifest = manifest_store.load_and_verify(source_replay_id)
    image_ref = image_resolver.require_digest(manifest.image_digest)
    handle = await scheduler.create(new_execution_id, image_ref, limits)
    removed = False
    await scheduler.wait_until_ready(handle)
    await artifact_transfer.upload(handle, manifest)
    await runtime_client.replay_submit(
        handle,
        ReplayRequest(
            execution_id=new_execution_id,
            replay_run_id=replay_run_id,
            source_replay_id=source_replay_id,
            mode=ReplayMode.STRICT,
            manifest_relative_path="manifest.json",
        ),
    )
    trajectory = await collect_complete_trajectory(handle)
    comparison = comparator.compare(manifest, trajectory)
except Exception as exc:
    failure = exc
finally:
    if handle is not None:
        try:
            await scheduler.destroy(handle)
            removed = True
        except Exception as cleanup_exc:
            removed = False
            failure = cleanup_exc

result = replay_result_factory.build(
    replay_run_id=replay_run_id,
    source_replay_id=source_replay_id,
    comparison=comparison,
    failure=failure,
    container_removed=removed,
)
replay_store.commit_run(replay_run_id, result, trajectory)
return result
```

任何失败都不得跳过容器清理。最终 `ReplayResult` 和 run 目录必须在清理尝试完成后提交，确保 `container_removed` 反映真实结果；清理失败必须覆盖原本的 matched 状态并标记基础设施失败，同时保留先前分歧诊断。

`image_resolver.require_digest()` 必须使用第一周调度器已经记录的实际不可变镜像标识；`image_ref` 仅用于定位本地镜像，不允许 strict 重放只依赖可漂移 tag。对于本地未推送镜像，可以使用 Docker image ID；对于仓库镜像，优先使用 repo digest，并在 Manifest 中明确其类型。

## 16. 分支注入规则

第一版支持：

| 注入类型 | 作用 |
|---|---|
| `prompt_replace` | 替换检查点后的当前用户 Prompt |
| `prompt_append` | 向当前 Prompt 追加受控文本 |
| `model_decision_replace` | 替换下一条结构化模型决策 |
| `tool_result_replace` | 替换下一条受控工具结果 |

优先级分为：

- 核心必交付：`prompt_replace`、`prompt_append` + `live_and_record` 后缀。
- 扩展交付：`model_decision_replace`、`tool_result_replace` + `strict_with_replacements`。扩展交付不得阻塞核心 strict 重放闭环。

限制：

- 注入内容必须经过协议大小限制和脱敏规则。
- 只能注入检查点声明允许的字段。
- 所有注入都必须重新经过 schema 校验和工具安全策略；`model_decision_replace` 不得绕过工具策略层。
- 注入后从该检查点起重新计算全部摘要。
- 父轨迹后缀只能作为比较基线，不能复制为子轨迹结果。
- 子轨迹必须使用新的 execution ID、trajectory ID 和 replay ID。
- 必须记录注入类型、内容摘要、操作者和创建时间。
- Prompt 发生变化后，默认使用 `live_and_record` 重新生成后缀；禁止继续无条件消费父轨迹的 RecordedDecisionModel。
- `strict_with_replacements` 必须提供从注入点到终止点所需的完整替代决策，或在第一处缺失决策时明确分歧。
- `tool_result_replace` 仅允许用于无副作用工具，或同时提供经过校验的受控状态增量；禁止只修改返回值而保留不匹配的环境状态。

## 19. 测试计划

### 19.1 单元测试

- Canonical JSON 对键顺序不敏感。
- Unicode NFC 规范化稳定。
- NaN、Infinity 和负零被拒绝，非整数按约定转换为定点字符串。
- `ArtifactRef.relative_path` 拒绝绝对路径、盘符、UNC、反斜杠和 `..`；普通 Prompt 字符串中的路径文本不由 canonical JSON 拒绝。
- 相同对象产生相同 SHA-256。
- Manifest 自身摘要计算与篡改检测。
- Manifest schema 校验。
- ArtifactRef 大小与摘要校验。
- Agent 状态导出/恢复。
- 虚拟文件系统导出/恢复。
- Mock API 消费索引恢复。
- PRNG 状态或消费计数恢复。
- RecordedDecisionModel 顺序消费。
- 模型决策和工具交互游标从检查点正确恢复。
- 模型输入摘要不匹配时失败。
- execute_and_verify 工具摘要比较。
- stub_response 参数摘要比较。
- 事件规范化排除时间戳和执行 ID。
- 重放审计事件不进入行为摘要，recorded/replayed 事件映射为同一行为语义。
- Comparator 定位首个分歧序号。
- 检查点可恢复性判定。
- 分支注入白名单。
- tar 路径穿越和符号链接拒绝。

### 19.2 集成测试

- 一次执行生成完整 ReplayManifest。
- Manifest 可定位全部产物。
- ReplayManifest 不保存运行 `mode`，strict/live 只由本次 ReplayRequest 决定。
- strict 重放不调用 FakeChatModel。
- strict 重放可恢复虚拟文件内容。
- strict 重放可恢复 Mock API 状态。
- 工具 execute_and_verify 成功。
- 工具结果变更时产生分歧。
- live 重放允许重新执行 Fake 模型。
- live 新工具请求标记 `replay_diverged`。
- 检查点恢复后节点和逻辑时间正确。
- Prompt 注入后使用 live-and-record 产生新的子轨迹和子 Manifest。
- 子轨迹通过父前缀引用和新后缀构成，不复制父轨迹旧后缀。
- 缺失产物时不创建或及时清理容器。
- 连续多次重放分别产生独立 replay run ID 和结果目录。

## 20. 第二周验收标准

### 20.1 录制与 Manifest

- [ ] 一次正常执行可以生成版本化 ReplayManifest。
- [ ] Manifest 记录实际镜像摘要、Agent 版本、Runtime 版本和协议版本。
- [ ] Manifest 记录状态编码、工具注册表、策略、规范化和 canonical JSON 版本。
- [ ] Manifest 能通过 ArtifactRef 定位所有必要产物。
- [ ] Prompt 原文或其受控产物可由 Manifest 定位，`prompt_digest` 可独立校验。
- [ ] 所有产物写入前后均校验 SHA-256。
- [ ] Manifest 自身摘要可校验，提交后不可原地修改。
- [ ] Manifest 和产物中不包含宿主机绝对路径或明文密钥。

### 20.2 strict/live 重放

- [ ] strict 模式不重新调用真实模型或 FakeChatModel。
- [ ] strict 模式按顺序注入已录制结构化模型决策。
- [ ] 工具逐交互支持 execute_and_verify 和 stub_response，并正确恢复副作用状态。
- [ ] 同一轨迹 strict 重放 10 次，每次行为轨迹摘要均与源轨迹一致，对齐检查点状态摘要一致，且最终受支持状态摘要一致。
- [ ] 重放审计事件完整保留，但不污染行为轨迹摘要。
- [ ] live 模式重新执行模型，但固定初始环境和已声明工具响应。
- [ ] live 模式出现未录制请求时明确标记分歧。
- [ ] 缺少产物、摘要不匹配或镜像缺失时明确失败。

### 20.3 检查点与分支

- [ ] 模型、节点和工具稳定边界可生成检查点。
- [ ] 每个检查点记录状态摘要和可恢复性。
- [ ] 每个可恢复检查点记录状态编码版本、恢复入口、模型/工具消费游标和已登记随机源状态。
- [ ] 可从任意已记录且可恢复的稳定检查点恢复。
- [ ] 可注入新 Prompt，并默认以 live-and-record 模式生成新轨迹和子 Manifest。
- [ ] 子轨迹记录父 replay、父 trajectory、fork sequence 和 injection digest。
- [ ] `fork_sequence` 明确表示最后一个继承的父行为事件，子轨迹不复制父轨迹旧后缀。
- [ ] 注入后的后续事件和状态摘要全部重新计算。

### 20.4 安全与工程

- [ ] 重放容器继续满足第一周全部安全基线。
- [ ] Artifact tar 拒绝路径穿越、绝对路径、设备文件和符号链接。
- [ ] 所有成功、失败、分歧、取消和崩溃路径均无遗留容器。
- [ ] 单元、集成和真实 Docker 重放测试通过。
- [ ] 第一周基线测试套件全部继续通过（当前共 49 项）。
- [ ] README 包含录制、strict/live 重放、分支和故障排查命令。

## 21. 风险与处理策略

| 风险 | 影响 | 处理策略 |
|---|---|---|
| 将重放实现为重新请求模型 | 无法稳定复现 | strict 使用 RecordedDecisionModel，禁止回退 live |
| 对整个 Python 对象做 pickle | 安全风险和版本不兼容 | 仅序列化白名单 JSON 状态 |
| 动态字段进入摘要 | 同一轨迹摘要不稳定 | 固定 normalization_version，排除时间戳和运行 ID |
| 重放控制事件进入行为摘要 | 原轨迹与重放轨迹必然不一致 | 行为轨迹与 ReplayAuditStore 分开，并对 recorded/replayed 事件做语义投影 |
| 工具桩响应与当前参数不匹配 | 错误复用历史结果 | 参数摘要必须一致，否则 replay_diverged |
| 全局工具策略覆盖逐交互策略 | 一条轨迹无法混合安全执行和桩回放 | 每条 RecordedToolInteraction 固化 replay_mode |
| write_file 副作用重复执行 | 状态偏移 | 恢复初始 VFS 后执行并比较前后摘要 |
| 只恢复初始 seed | 从中段检查点恢复后随机序列偏移 | 导出 PRNG 当前状态或确定性消费计数 |
| 检查点位于不稳定边界 | 无法安全恢复 | 只允许协议声明的稳定边界 |
| 检查点缺少消费游标 | 重复消费或跳过模型决策/工具响应 | 状态信封记录下一模型和工具索引及 resume_phase |
| Manifest 指向被修改文件 | 重放证据不可信 | 每次读取重新计算 SHA-256 |
| tar 产物路径穿越 | 写出容器工作区 | 严格校验成员类型和规范化路径 |
| 镜像 tag 漂移 | 环境不可复现 | strict 使用实际 image digest，不只使用 tag |
| 分支复制父轨迹后缀 | 子轨迹伪造 | fork 后所有后缀必须重新执行和计算摘要 |
| Prompt 变异后继续消费父决策 | 输入摘要立即分歧，无法形成新分支 | 默认 live-and-record；strict 分支要求完整替代决策 |
| Manifest 被修改但产物摘要未变 | 重放包语义被替换 | 对 Manifest payload 单独计算并校验摘要，提交后不可变 |
| 产物体积持续增长 | 磁盘压力 | 内容寻址去重、大小限制和保留策略 |
| Docker Exec 任意命令风险 | 扩大控制面权限 | 继续只允许固定 RPC helper 和固定 Artifact API |

## 23. 第二周完成定义

当以下闭环稳定通过时，第二周任务视为完成：

```text
运行一条 Agent 测试
  → 记录模型决策、工具响应和稳定检查点
  → 导出初始状态与虚拟环境快照
  → 生成包含全部摘要的 ReplayManifest
  → 从 Manifest 创建全新无网容器
  → strict 模式恢复状态并注入历史决策
  → 重新生成行为轨迹并另行记录重放审计事件
  → 规范化行为轨迹摘要和最终受支持状态摘要与历史一致
  → strict 重放连续 10 次均与源轨迹一致，且每次具有独立 replay run ID
  → 选择一个可恢复检查点
  → 注入新 Prompt
  → 以 live-and-record 重新执行后缀
  → 生成具有父前缀引用、全新后缀和父子关系的新分支轨迹及子 Manifest
  → 所有产物可定位、可校验
  → 所有容器被删除且无遗留资源
```

第二周最终交付物应当是一套基于显式逻辑状态、内容寻址产物和稳定检查点的可验证重放系统，而不是依赖模型“碰巧再次生成相同输出”的随机复现工具。
