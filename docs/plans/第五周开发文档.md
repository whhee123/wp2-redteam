## 1. 前置基线与方向约束

第五周直接复用以下公开能力：

1. `RedTeamExecutionEngine.run_test_case()` 负责单个 `TestCase` 的完整容器生命周期。
2. `ReplayEngine.fork()` 可以执行结构化 Fork 候选。
3. `CoverageInputResolver` 将已提交轨迹或 Replay 产物转换为 `CoverageInput`。
4. `CoverageStore.evaluate()` 幂等计算行为覆盖、风险覆盖、风险深度和行为-风险关联。
5. `CoverageStore.snapshot()` 提供 Campaign 累计覆盖快照和完整 `risk_depths`。
6. `MutationFeedbackBuilder.build()` 将覆盖率、风险可达集和变异历史转换为反馈。
7. `SemanticMutator.mutate()` 生成去重、排序并持久化的 `MutationCandidate`。
8. `MutationStore` 提供候选、批次和历史统计公开接口。
9. `MutationCandidate.path_signature` 已在生成与统计路径之间统一。
10. `FakeChatModel` 和 `RuleBasedMutationProvider` 保持确定性，用于 CI 和无服务器环境。
11. Agent 和 Mutator 均已预留本地 Ollama 配置及 `model_digest`。

最终部署方向不变：

```text
服务器宿主机
  ├── Fuzzer Orchestrator
  ├── SeedPool / EnergyScheduler / CorpusManager
  ├── MutationProvider（本地 RuleBased 或本地 Ollama）
  ├── Coverage Engine
  ├── Campaign Store
  └── Docker Scheduler
          |
          v
  每个候选进入独立、限权、默认无公网的 Agent 容器
          |
          v
  仅在显式配置时访问服务器本地模型端点
```

当前服务器尚未申请下来，因此第五周验收分三层：

- **确定性功能验收**：Fake Agent + RuleBased Mutator，本机和 CI 必须完成。
- **本机基础设施耐久验收**：Docker Desktop 上完成并发、暂停/恢复、故障注入和资源检查。
- **服务器语义与容量验收**：真实本地模型、1 至 2 个 Target Profile、GPU 竞争和 24 小时有效 Fuzzing，标记为 server-gated。

不能把 RuleBased Provider 的结构变体描述为真实语义探索，也不能把重复执行探针产生的数据计入覆盖增长。

## 4. 总体架构

### 4.1 组件关系

```text
Campaign CLI
    |
    v
FuzzingEngine / Coordinator（唯一状态写者）
    |
    ├── FuzzerStore
    ├── SeedPool
    ├── EnergyScheduler
    ├── CorpusManager
    ├── MutationFeedbackBuilder
    ├── SemanticMutator
    ├── CoverageStore
    └── MutationStore
    |
    +--> 持久 WorkItem
             |
             v
       bounded execution_queue
             |
      +------+------+------+
      | Worker 1 | Worker 2 | ... Worker N
      +------+------+------+
             |
             v
       RedTeamExecutionEngine
       或 ReplayEngine.fork()
             |
             v
       bounded result_queue
             |
             v
Coordinator 单写者提交
  -> 轨迹恢复/解析
  -> CoverageStore.evaluate()
  -> CorpusPolicy.evaluate()
  -> SeedPool 更新/晋升
  -> WorkItem COMMITTED
  -> CampaignSnapshot
```

### 4.2 并发边界

第五周只并发昂贵的 Docker 执行：

- Mutator 由 Coordinator 调用，可与已运行的执行任务交叠。
- Docker 执行由 N 个 Worker 并发。
- CoverageStore、MutationStore、FuzzerStore 只由 Coordinator 调用。
- Worker 只返回序列化 Outcome，不写 SQLite。
- 每个 Worker 使用独立 Docker client、Scheduler 和 RuntimeClient。

这避免同一 SQLite 连接跨线程，也避免不必要的锁竞争。未来 Coverage 变重时可并发纯计算，但数据库提交仍保持单写者。

### 4.3 数据流

```text
初始 TestCase
  -> bootstrap 执行与覆盖
  -> SeedPool ACTIVE
  -> SeedPool.select()
  -> EnergyScheduler.assign()
  -> MutationFeedbackBuilder
  -> SemanticMutator.mutate(count=energy)
  -> MutationCandidate
  -> WorkItem QUEUED
  -> Worker 隔离执行
  -> CandidateExecutionOutcome
  -> CoverageStore.evaluate()
  -> CorpusPolicy
  -> Seed 晋升/冷却/淘汰
  -> 下一轮
```

## 6. 核心数据契约

所有模型继承 `FuzzerContract`，使用 Pydantic v2 和 `extra="forbid"`。时间使用 UTC ISO 8601，内容摘要复用 Canonical JSON + SHA-256。

### 6.1 状态枚举

```python
class CampaignStatus(StrEnum):
    CREATED = "created"
    BOOTSTRAPPING = "bootstrapping"
    RUNNING = "running"
    PAUSE_REQUESTED = "pause_requested"
    PAUSED = "paused"
    STOP_REQUESTED = "stop_requested"
    COMPLETED = "completed"
    FAILED = "failed"


class CampaignStopReason(StrEnum):
    BUDGET_EXHAUSTED = "budget_exhausted"
    STAGNATED = "stagnated"
    USER_REQUESTED = "user_requested"
    NO_ELIGIBLE_SEEDS = "no_eligible_seeds"
    SYSTEMIC_INFRASTRUCTURE_FAILURE = "systemic_infrastructure_failure"


class SeedStatus(StrEnum):
    PENDING = "pending"
    ACTIVE = "active"
    COOLED = "cooled"
    RETIRED = "retired"
    QUARANTINED = "quarantined"


class WorkSourceKind(StrEnum):
    INITIAL_CASE = "initial_case"
    MUTATION = "mutation"
    FORK = "fork"
    SOAK_PROBE = "soak_probe"


class WorkSourceRef(FuzzerContract):
    kind: WorkSourceKind
    case_id: str | None = None
    candidate_id: Digest | None = None
    probe_id: str | None = None

    @model_validator(mode="after")
    def validate_source(self) -> WorkSourceRef:
        if self.kind == WorkSourceKind.INITIAL_CASE:
            if not self.case_id or self.candidate_id or self.probe_id:
                raise ValueError("initial case requires only case_id")
        elif self.kind in {WorkSourceKind.MUTATION, WorkSourceKind.FORK}:
            if not self.candidate_id or self.case_id or self.probe_id:
                raise ValueError("mutation/fork requires only candidate_id")
        elif not self.probe_id or not self.case_id or self.candidate_id:
            raise ValueError("soak probe requires probe_id and case_id")
        return self


class WorkItemStatus(StrEnum):
    QUEUED = "queued"
    LEASED = "leased"
    EXECUTED = "executed"
    COMMITTED = "committed"
    RETRY_WAIT = "retry_wait"
    FAILED = "failed"
    DEAD_LETTER = "dead_letter"
    SKIPPED = "skipped"


class FailureKind(StrEnum):
    CASE_FAILURE = "case_failure"
    TRANSIENT_INFRASTRUCTURE = "transient_infrastructure"
    SYSTEMIC_INFRASTRUCTURE = "systemic_infrastructure"
    INTEGRITY_FAILURE = "integrity_failure"
    CANCELLED = "cancelled"


class CorpusReason(StrEnum):
    NEW_BEHAVIOR = "new_behavior"
    NEW_RISK_CATEGORY = "new_risk_category"
    RISK_DEPTH_INCREASE = "risk_depth_increase"
    NEW_BEHAVIOR_RISK_LINK = "new_behavior_risk_link"
    POLICY_VIOLATION = "policy_violation"
    DIVERGENT_SECURITY_OUTCOME = "divergent_security_outcome"
```

### 6.5 WorkItem

```python
class WorkItem(FuzzerContract):
    work_item_id: Digest
    campaign_id: str
    source: WorkSourceRef
    parent_seed_id: Digest | None = None
    priority: float
    status: WorkItemStatus
    created_iteration: int
    dispatch_sequence: int | None = None

    attempt: int = 0
    execution_id: str | None = None
    lease_owner: str | None = None
    lease_token_digest: Digest | None = None
    lease_expires_at: datetime | None = None
    retry_not_before: datetime | None = None

    trajectory_id: str | None = None
    trajectory_path: str | None = None
    replay_id: str | None = None
    coverage_result_digest: Digest | None = None
    corpus_entry_id: Digest | None = None
    failure_kind: FailureKind | None = None
    error_code: str | None = None
```

`work_item_id` 由 Campaign、规范化 `source` 和逻辑版本生成。相同 Mutation/Fork Candidate 在一个 Campaign 中只能有一个逻辑 WorkItem；每次 Soak Probe 运行使用独立 `probe_id`。`dispatch_sequence` 在 `QUEUED -> LEASED` 时原子分配，QUEUED/SKIPPED 且从未租约的任务不占序号；Retry 保留原序号。

WorkItem 的 model validator 必须执行以下不变量：

- INITIAL_CASE 和 SOAK_PROBE 的 `parent_seed_id` 可以为空；MUTATION/FORK 必须有 `parent_seed_id`。
- 首次 QUEUED 时 `dispatch_sequence` 为空；成功 Lease 后为正整数且不可修改。
- RETRY_WAIT 及 retry 后的 QUEUED 保留原 `dispatch_sequence`，下一次 Lease 不重新分配。
- EXECUTED、COMMITTED、FAILED、DEAD_LETTER 必须具有 `dispatch_sequence`。
- 从未 Lease 就因 Budget 跳过的 SKIPPED 可以没有序号；Lease 后取消形成的 SKIPPED 必须保留序号。
- `lease_owner`、`lease_token_digest`、`lease_expires_at` 必须同时出现或同时为空。

### 6.8 CampaignSnapshot

快照至少包含：

- Campaign 状态和停止原因。
- iteration 和有效运行时长。
- ACTIVE/COOLED/RETIRED Seed 数。
- QUEUED/LEASED/COMMITTED/FAILED/DEAD_LETTER WorkItem 数。
- Corpus 大小。
- 成功、Case 失败、基础设施失败和 retry 数。
- 未清理容器数。
- 行为特征、行为档案和风险类别总数。
- applicable intent/behavior/impact coverage。
- 新行为每小时、风险深度增量每小时。
- execution P50/P95。
- 队列长度和当前 Worker 数。

```yaml
    every_committed_items: 10
    every_seconds: 60

  shutdown:
    graceful_timeout_seconds: 180
    cleanup_timeout_seconds: 60
```

### 7.2 校验

- 至少设置一个有限 Budget。
- `min_energy <= base_energy <= max_energy`。
- Worker 数至少为 1，Queue 大小不小于 Worker 数。
- `max_pending_work_items >= execution_queue_size`。
- Lease 时间至少为 heartbeat 的 3 倍。
- retry 不超过 5 次。
- 系统性错误阈值位于 `(0, 1]`。
- Campaign ID 满足现有路径安全规则。
- Fuzzer、Mutation、Coverage 使用相同 Campaign ID 或显式一一映射。
- Fake Agent 模式禁止启用 Ollama 网络。
- Ollama Agent 必须配置受控网络、模型名和模型 digest。
- `deterministic_rounds` 要求 `max_feedback_lag_work_items == 0`。
- `throughput` 要求 `max_feedback_lag_work_items >= sandbox_workers` 且不超过 `max_pending_work_items`。

## 8. SeedPool

### 8.1 Bootstrap

初始模板必须先执行并计算基线：

```text
initial_case_ids
  -> SeedRecord(status=PENDING)
  -> WorkItem(source=INITIAL_CASE)
  -> 标准隔离执行
  -> CoverageStore.evaluate()
  -> 写入 coverage/profile/risk
  -> PENDING -> ACTIVE
  -> Campaign RUNNING
```

恢复时如已有相同 digest 的覆盖结果则复用。

### 8.2 公平选种

使用持久化 Weighted Virtual Runtime：

```text
从 ACTIVE 且未 cooldown 的种子中选择：
  (virtual_runtime, seed_id) 最小者

选中并计算 EnergyDecision 后：
  virtual_runtime += 1 / max(1, assigned_energy)
```

高能量种子通过较小的 virtual runtime 增量获得更多后续机会，低能量种子不会永久饥饿，`seed_id` 提供稳定 tie-break。选种本身不依赖未定义的预估能量；EnergyDecision 在选中后使用同一持久状态计算。新晋升 Seed 的 virtual runtime 取当前 ACTIVE Seed 的最小值。

### 8.3 生命周期

- PENDING：尚未完成 Bootstrap，不允许普通选择和变异。
- ACTIVE：允许选择和变异。
- COOLED：连续无增益，暂时跳过。
- RETIRED：达到最大深度或寿命终点。
- QUARANTINED：稳定非法结构、不可恢复完整性问题或确定性异常。

冷却到期后回到 ACTIVE，`consecutive_no_gain` 衰减一半。只有真实新增行为、风险类别或风险深度才清零。

### 8.4 晋升条件

- 轨迹和 CoverageResult 均已提交。
- 不是基础设施错误。
- 满足 CorpusPolicy。
- 未达到最大 Mutation Depth。
- Prompt/Fork 血缘完整。
- seed ID 不冲突。
- Fork 能恢复实际 Prompt 和 replay 引用。

Corpus Entry 可以被保留但不继续作为 Seed，例如已经达到最大深度。

## 9. 能量调度

### 9.1 energy-v1

```text
RawEnergy =
    BaseEnergy
  * NoveltyFactor
  * RiskGapFactor
  * RarityFactor
  * StabilityFactor
  * StagnationFactor
  * DepthFactor

AssignedEnergy = clamp(round(RawEnergy), MinEnergy, MaxEnergy)
```

```text
NoveltyFactor =
  1 + novelty_weight * clamp(parent_combined_delta, 0, 1)

seed_target_gaps =
  feedback.risk_gaps whose category_id is in seed.case.target_risks
  or all feedback.risk_gaps when the intersection is empty

weighted_gap =
  sum(gap_ratio * report_weight * schedule_weight)
  / max(epsilon, sum(report_weight * schedule_weight))

RiskGapFactor =
  1 + risk_gap_weight * clamp(weighted_gap, 0, 1)

profile_frequency =
  FuzzerStore.profile_execution_count(seed.behavior_profile_hash)
  or 1 for a Seed without a committed profile

RarityFactor =
  1 + rarity_weight / sqrt(max(1, profile_frequency))

StabilityFactor =
  1.0                                  if attempts == 0
  clamp(successes / attempts, 0.25, 1) otherwise

StagnationFactor =
  max(
    minimum_stagnation_factor,
    stagnation_decay
      ** floor(consecutive_no_gain / stagnation_decay_interval)
  )

DepthFactor =
  max(0.25, 1 - mutation_depth / max_mutation_depth)
```

约束：

- Novelty 只使用真实 Coverage，不使用 RuleBased `expected_novelty`。
- RiskGap 直接使用 MutationFeedback 中已按 Campaign 可达集构造的 RiskGap。
- Rarity 使用 FuzzerStore 对所有已 COMMITTED Observation 幂等维护的 profile 频次，不调用 CoverageStore 私有查询，也不使用原始 Prompt 字符差异。
- attempts/successes 指该 Seed 已执行子任务中的正常终态数和成功数；基础设施 attempt 单独统计。
- Docker 或模型服务故障不降低 Seed Stability。
- 时间不得进入能量公式，只用于租约和 Budget。
- 相同 Manifest、Snapshot、Seed、History、Config 和 iteration 必须产生相同 Decision。

Cold Start Seed 使用全局 RiskGap，其余因子为 1.0，并优先完成 Bootstrap。

### 9.2 调度数据的公开来源

第五周只通过以下来源计算 Energy：

- `MutationFeedback.risk_gaps`：RiskGap 和 schedule weight。
- `SeedRecord.coverage_result_digest` 对应的 CoverageResult：父 Seed 真实增量。
- `FuzzerStore.profile_execution_count()`：行为档案频次。
- `FuzzerStore` 中的 Seed 选择、子任务成功和无增益计数。
- `CoverageStore.snapshot()`：累计覆盖与快照 digest。
- `CoverageStore.schedule_weights()`：现有公开方法，正式加入第四周交接清单。

不调用 `CoverageStore.all_profiles()`，也不读取 coverage.db 私有表。

## 10. Corpus 策略

`coverage-corpus-v1` 满足任一条件即可保留：

1. `new_behavior_count > 0`。
2. `new_risk_count > 0`。
3. `risk_depth_changes` 非空。
4. 存在非 `known_pair` 的 BehaviorRiskLink。
5. RuleBasedScorer 给出 `violation`。
6. 相同行为档案/path signature 出现新的安全 verdict。

不保留：

- 仅文字预计新颖。
- 仅字符差异较大但真实行为和风险无变化。
- 基础设施错误。
- 无终态轨迹。
- Coverage 完整性失败。
- 重复 trajectory。

Corpus 是证据集合，SeedPool 是继续消耗预算的工作集合，二者不得合并。

维护索引：

```text
(behavior_profile_hash, target_risks, path_signature)
  -> observed verdict set
```

同一索引首次出现不同 verdict 时记录 `DIVERGENT_SECURITY_OUTCOME`。

## 11. WorkItem、租约与重试

### 11.1 状态机

```text
QUEUED
  -> LEASED
      -> EXECUTED
          -> COMMITTED
      -> RETRY_WAIT -> QUEUED
      -> FAILED
      -> DEAD_LETTER
  -> SKIPPED
```

非法跳转抛 `FuzzerIntegrityError`。不同 lease token 不能完成同一任务，COMMITTED 不得回到 QUEUED。

### 11.2 租约

Coordinator 在放入内存 Queue 前，通过单个事务完成 `QUEUED -> LEASED`、分配下一个连续 `dispatch_sequence`、生成 execution ID 和写入 Lease。Lease 保存 worker、token digest、expires_at、attempt 和 execution ID。QUEUED WorkItem 可以按 priority 选择，不受创建顺序约束；只有成功 Lease 的任务进入确定性提交序列。Worker 不写数据库，Coordinator 周期续约。

重启时：

- 未过期 Lease 先对账产物。
- 过期且无完整产物的任务回到 QUEUED/RETRY_WAIT。
- 有完整轨迹的任务转 EXECUTED，继续 Coverage。
- Coverage 已提交但 Fuzzer commit 未完成时，重新调用幂等 evaluate，再完成提交。

### 11.3 确定性 execution ID

现有随机 UUID 不足以可靠恢复。扩展接口：

```python
async def run_test_case(
    self,
    case: TestCase,
    *,
    execution_id: str | None = None,
    run_context: SandboxRunContext | None = None,
) -> RunOutcome:
    ...
```

Fuzzer 使用：

```text
"fuzz-" + sha256(campaign_id | work_item_id | attempt)[:24]
```

普通 CLI 不传 ID，行为不变。retry 增加 attempt，不覆盖前次 `.partial`。恢复同一 attempt 时先检查最终轨迹，不盲目重跑。

所有显式 execution ID 先通过共享 `validate_execution_id()`：

- 通用格式：`^[A-Za-z0-9][A-Za-z0-9._-]{0,127}$`。
- 拒绝 `.`、`..`、斜杠、反斜杠、冒号、控制字符和路径分隔符。
- Fuzzer 派生 ID 进一步要求 `^fuzz-[0-9a-f]{24}$`。
- TrajectoryStore 在拼接 `partial_path` 和 `final_path` 前再次验证，不能只信任上层。
- 相同 execution ID 已有 final 轨迹时由 RecoveryManager 走恢复路径；普通新执行仍触发完整性错误。

### 11.4 错误分类

- **Case Failure**：Agent 错误终态、step limit、工具拒绝。默认不重试。
- **Transient Infrastructure**：偶发 readiness、Docker API、通信中断。有限指数退避。
- **Systemic Infrastructure**：daemon 不可用、镜像/模型 digest 冲突、网络缺失。自动安全暂停。
- **Integrity Failure**：ID payload 冲突、轨迹摘要错误、非法状态。进入 DEAD_LETTER 并默认暂停。

退避：

```text
delay = min(initial_backoff * 2 ** attempt, max_backoff)
```

基础设施 retry 计入执行尝试，但不计 Seed 无增益。

## 13. 并发与背压

Coordinator 只有满足以下条件才生成：

- Campaign 为 RUNNING。
- Budget 未耗尽。
- pending 小于上限。
- Queue 有容量。
- 存在可选 Seed。
- 系统性故障断路器关闭。

生成数：

```text
requested = min(
  assigned_energy,
  max_pending_work_items - pending_work_items,
  remaining_execution_budget
)
```

不能无限生成再排队，否则 MutationStore 的 accepted 统计会被未执行积压污染。

启动前估算：

```text
required_memory = workers * container_memory
required_cpu = workers * container_nano_cpus
```

超出用户声明主机预算时拒绝启动。Docker Desktop 自动探测只给提示，不覆盖显式配置。

### 13.1 确定性提交顺序

Docker Worker 可以乱序完成，但 CoverageStore 的新增覆盖归属与提交先后有关。因此 Coordinator 必须按 `dispatch_sequence` 严格递增提交 Coverage、Corpus 和 Seed 更新：

```text
Worker 2 先完成 sequence=11
  -> Outcome 持久化为 EXECUTED，进入 result buffer
Worker 1 后完成 sequence=10
  -> 提交 sequence=10
  -> 再提交 sequence=11
```

结果缓冲区受 `max_pending_work_items` 约束。较早任务进入 retry 时，后续 Coverage 提交等待该任务产生终态结果，形成有界的 head-of-line blocking。Case Failure、DEAD_LETTER 或 SKIPPED 都是可推进序号的终态。

该规则保证相同 Dispatch 序列下，新行为、新风险和 Corpus 归属不受容器完成时序影响。恢复时同样从最小未提交 `dispatch_sequence` 继续。`dispatch_sequence` 只在 Lease 事务中分配，因此未派发的低优先级任务不会阻塞已经执行的任务。

### 13.2 调度模式

配置必须显式选择：

- `deterministic_rounds`：默认用于 CI、回归和可复现实验。一个 wave 中可并发执行多个 WorkItem，但只有该 wave 全部产生可提交终态并按序提交后，才能读取新 Snapshot、选择 Seed 和生成下一 wave。`max_feedback_lag_work_items` 必须为 0。
- `throughput`：用于服务器容量和长时间探索。允许在已有执行任务时继续生成，但未提交反馈滞后不得超过 `max_feedback_lag_work_items`。每个 MutationBatch 必须保存实际使用的 coverage snapshot digest、mutation history digest 和 Seed state digest。

`deterministic_rounds` 保证同一配置和故障脚本下的 Seed 选择、MutationBatch 和覆盖归属可复现。`throughput` 不承诺整个 Campaign 的候选顺序逐字节一致，但所有决策输入都可审计和单项重放。

## 14. 主循环与提交顺序

### 14.1 主循环

```python
async def run(self):
    await recovery.reconcile()
    await bootstrap_if_needed()
    await store.transition_to_running()
    workers = start_execution_workers()

    try:
        while True:
            await drain_worker_results()

            if pause_or_stop_requested():
                break
            if budget.exhausted():
                await complete(BUDGET_EXHAUSTED)
                break
            if stagnated():
                await complete(STAGNATED)
                break
            if circuit_breaker.open:
                await request_pause(SYSTEMIC_INFRASTRUCTURE_FAILURE)
                break

            await dispatch_queued_work()

            if scheduling.requires_wave_barrier() and work_in_flight():
                await wait_for_result()
                continue

            if capacity_available() and scheduling.feedback_lag_within_limit():
                seed = seed_pool.select()
                if seed is None:
                    if work_in_flight():
                        await wait_for_result()
                        continue
                    if seed_pool.has_cooled_seeds():
                        await store.advance_iteration_to(
                            seed_pool.next_cooldown_iteration()
                        )
                        await seed_pool.reactivate_due_seeds()
                        continue
                    await complete(NO_ELIGIBLE_SEEDS)
                    break

                snapshot = coverage_store.snapshot(include_heatmap=False)
                feedback = feedback_builder.build(
                    seed.to_mutation_seed(),
                    snapshot,
                    schedule_weights=coverage_store.schedule_weights(),
                    history=mutation_store.snapshot(),
                )
                energy = energy_scheduler.assign(seed, feedback, snapshot)
                batch = await mutator.mutate(
                    seed.to_mutation_seed(),
                    feedback,
                    count=limit_by_capacity(energy.assigned_energy),
                    random_seed=derive_iteration_seed(),
                )
                dispatch = await store.enqueue_new_candidates(
                    batch.accepted,
                    seed,
                    energy,
                )
                await seed_pool.record_generation_result(
                    seed.seed_id,
                    new_work_items=dispatch.created_count,
                    batch_exhausted=batch.exhausted,
                    already_generated=batch.already_generated,
                )
                await store.advance_iteration()

            await wait_for_progress_or_timer()
    finally:
        await graceful_shutdown(workers)
```

### 14.2 跨 Store 一致性

不存在跨 trajectory、coverage.db、mutation.db、fuzzer.db 的原子事务。固定顺序：

1. Execution Engine 原子提交轨迹。
2. FuzzerStore 标记 WorkItem EXECUTED，记录轨迹引用。
3. CoverageStore 幂等 evaluate。
4. FuzzerStore 单事务保存 Observation、Corpus、Seed 更新、父 Seed 统计并标记 COMMITTED。
5. 定期原子写 Snapshot JSON。

系统保证逻辑结果幂等提交，不声称 Docker 执行绝对 exactly-once。

### 14.3 Budget

- max executions 计真实执行 attempt，不计纯恢复提交。
- transient retry 计入 attempt。
- bootstrap 默认计入 Budget。
- Budget 耗尽时未派发 WorkItem 标记 SKIPPED。
- 到达时间上限后停止生成和派发，等待 grace period。
- 运行时间使用单调时钟并叠加持久累计值。

### 14.4 无新任务与停滞

`enqueue_new_candidates()` 只为尚不存在逻辑 WorkItem 的 Candidate 创建任务，并返回 `created_count`、`duplicate_count` 和 `rejected_count`。

每次 Mutator 调用后都必须推进逻辑 iteration。若 `created_count == 0`：

1. 增加父 Seed 的 `consecutive_generation_no_progress`。
2. 达到 `generation_no_progress_threshold` 后将该 Seed COOLED。
3. 不增加 Coverage 无增益计数，因为没有发生真实执行。
4. 如果所有 Seed 暂时 COOLED，但尚未达到 Campaign stagnation window，则逻辑 iteration 快进到最近 cooldown 到期点并重新激活到期 Seed，不使用 CPU 忙等。
5. 如果一个完整 SeedPool sweep 均未产生新 WorkItem，则增加 Campaign 级 generation-no-progress 计数；达到 `stagnation_window` 且没有 QUEUED、LEASED、EXECUTED 或 RETRY_WAIT 任务时，以 STAGNATED 完成。
6. 如果仍有 in-flight WorkItem，则等待结果，不进入生成忙循环。

只要创建至少一个新 WorkItem，就把该 Seed 的 generation no-progress 计数清零。`batch.already_generated`、`batch.exhausted` 和“accepted 候选已经存在”都必须进入该判断。

## 15. 暂停、恢复与停止

### 15.1 Pause

1. 原子转 PAUSE_REQUESTED。
2. 停止生成和 Lease。
3. 等待活动任务至 graceful timeout。
4. 超时任务取消。
5. 使用 shield 保护容器 cleanup。
6. 未完成任务转 RETRY_WAIT/QUEUED。
7. 写 Snapshot。
8. 活动 Worker 为 0 且容器对账完成后转 PAUSED。

### 15.2 Resume

1. 读取 Manifest。
2. 校验锁定 digest。
3. 校验三个 Store 的 Campaign 映射。
4. 对账 EXECUTED、LEASED、RETRY_WAIT。
5. 扫描 final、partial 和 Replay 产物。
6. 精确清理本 Campaign 孤儿容器。
7. 回收过期 Lease。
8. 写 recovery audit。
9. 转 RUNNING。

### 15.3 Stop

Stop 不删除数据：

- 停止新任务。
- 完成或取消活动任务。
- 清理容器。
- 未执行任务标记 SKIPPED。
- 状态 COMPLETED，reason USER_REQUESTED。
- 保留数据库、轨迹、Replay 和 Corpus。

第五周不提供 delete CLI。

```text
new_behavior_per_hour = new behavior / RUNNING hours
risk_depth_gain_per_hour = sum(depth_gain) / RUNNING hours
interesting_rate = corpus promotions / committed work
unique_profile_rate = new profile hashes / committed work
```

PAUSED 时间和 Soak Probe 不进入增长率。

## 19. 24 小时测试分层

### 19.1 确定性闭环

Fake Agent + RuleBased Mutator，有限 Budget，多轮验证选种、能量、变异、执行、Coverage、Corpus 和再选种。风险范围继续使用当前 Fake 环境的 `risk-scope-week3.yaml`。

### 19.2 本机基础设施 Soak

RuleBased/Fake 的可达空间有限，正常 Fuzzer 可能提前 STAGNATED。增加显式 `soak_probe`：

- 使用锁定控制用例。
- 周期验证容器生命周期。
- 标记 `measurement_class="soak_probe"`。
- 不进入 Corpus、不晋升 Seed、不计覆盖增长。
- 与正常 fuzz work 分开统计。

本机 24 小时验收：

- 无持续增长的遗留容器。
- Pause/Resume 成功。
- 单任务错误不终止 Campaign。
- 三个数据库持续可查询。
- 内存、句柄、WAL 和临时文件无无界增长。
- 最终清理成功。

### 19.3 服务器语义 Campaign

服务器到位后：

- 锁定 Agent、Mutator 和镜像 digest。
- 使用 1 至 2 个 Target Profile。
- 无公网，容器只访问允许的本地模型端口。
- 统计 Schema 有效率、重复率、有效候选率、覆盖增长、GPU/CPU/内存和模型排队。
- 完成 24 小时有效 Fuzzing。
- 至少演练一次 Pause、进程退出和 Resume。
- 无遗留容器和未对账 Lease。

服务器验收前，报告必须写“功能与本机耐久性完成，真实模型容量验收待服务器”。

## 22. 安全要求

1. 所有 Initial Case、Mutation、Fork 和 Soak Probe 都必须进入对应的标准隔离执行路径。
2. 并发不放宽 CPU、内存、PID、只读根、capability 和 no-new-privileges。
3. Fake 模式保持 `network_mode="none"`。
4. Ollama 只能使用已标记的受控 bridge。
5. Mutator endpoint 与 Agent endpoint 分离。
6. Resume 不允许替换 image/model digest。
7. Worker 不执行候选文本中的宿主机命令。
8. 路径、命令、URL 均作为输入数据，不在 Orchestrator shell 解释。
9. 日志默认保存 digest。
10. 完整 Prompt 仅在显式导出时输出并给出警告。
11. 多 Campaign 清理按 label 精确匹配。
12. 取消和异常路径使用受保护 cleanup。
13. dead letter 不自动回到 SeedPool。
14. Soak Probe 不污染安全结论。
15. SQLite 使用绑定参数。
16. Campaign ID 和输出路径做逃逸校验。
17. 显式 execution ID 在 ExecutionEngine 和 TrajectoryStore 两层校验，不能包含任何路径语义。

## 23. 测试计划

### 23.1 单元测试

- 模型拒绝 extra 和非法状态，WorkSourceRef 四类联合约束分别覆盖。
- Campaign 状态转移矩阵。
- Manifest digest 冲突。
- Seed/Work ID 跨进程稳定，Initial Case/Mutation/Fork/Soak Probe identity 不冲突。
- Weighted Virtual Runtime 确定且无永久饥饿；全体暂时 COOLED 时安全快进而不误报 NO_ELIGIBLE_SEEDS。
- Energy 分量范围、确定性和边界。
- RuleBased expected novelty 不进入能量。
- 风险深度提升增加价值。
- 高频 profile 降低 rarity。
- 无增益触发衰减/冷却。
- 基础设施失败不计无增益。
- 最大深度退休。
- Corpus 六种理由。
- 纯文字差异不进入 Corpus。
- WorkItem 合法/非法跳转，PENDING Seed 不可选择，dispatch sequence 仅在 Lease 时连续分配。
- Lease token、过期回收。
- retry 退避和上限。
- 系统性错误断路器。
- Budget 不超发。
- 指标空样本/P50/P95。
- Soak Probe 隔离且不会调用 CoverageStore、CorpusPolicy 或 SeedPool 晋升。
- 空批次、already-generated 批次和全重复候选会推进 iteration，并最终冷却/停滞而不是忙循环。
- deterministic rounds 与 throughput 的反馈滞后约束。

### 23.2 集成测试

使用确定性内存 Executor：

```text
2 initial seeds
  -> bootstrap
  -> select + energy + mutate
  -> bounded queue
  -> 2 workers out-of-order finish
  -> single writer coverage/corpus
  -> promote seed
  -> next round
```

验证：

- 上一轮 Coverage 影响下一轮。
- 高价值子种子成为父种子。
- Queue 不越界。
- Worker 可以乱序完成，但 Coverage/Corpus 必须按 dispatch sequence 提交。
- Candidate 不重复创建逻辑任务；Initial Case 和每次 Soak Probe 使用合法且不同的 source identity。
- 崩溃在 trajectory 后/coverage 前可恢复。
- 崩溃在 coverage 后/fuzzer commit 前可恢复。
- Pause 不再生成和 Lease。
- Resume 不重复 Corpus。
- transient retry，case failure 不盲目 retry。
- systemic failure 自动暂停。
- SQLite 重开一致。
- CLI 全流程可用。

### 23.3 Docker 多轮 E2E

```text
创建 Campaign
  -> bootstrap 至少 2 Seed
  -> concurrency=2
  -> 至少 3 mutation iterations
  -> 至少 6 accepted candidates
  -> 独立容器执行
  -> CoverageResult 可查询
  -> 至少 1 Corpus Entry
  -> 至少 1 Seed 晋升并再次被选择
  -> Pause 并确认无 Campaign 容器
  -> Resume 后再提交至少 1 WorkItem
  -> Stop 并确认无容器/volume
```

不硬编码“覆盖一定高于父种子”，闭环状态和价值判定分别测试。

### 23.4 Docker 恢复 E2E

故障点：

1. Lease 后、容器前。
2. 容器运行期间终止 Orchestrator。
3. trajectory commit 后、Coverage 前。
4. Coverage 后、Fuzzer commit 前。
5. pause grace timeout。
6. destroy 首次失败、恢复清理成功。

验证 Lease 回收、完成轨迹不盲目重跑、Coverage/Corpus/Seed 统计幂等、孤儿清理且不影响其他 Campaign。

### 23.5 回归

- 第一至第四周非 Docker 测试全部通过。
- Mutation path signature 回归继续通过。
- run/record/replay/fork/coverage/mutate CLI 不变。
- 不重复增加已证明确定性的 strict replay 次数测试。
- 不增加当前没有真实状态副作用的 stub state-delta 测试。

## 24. 验收标准

### 24.1 闭环

1. Initial Seed 完成 Bootstrap。
2. Seed 获得可解释 EnergyDecision。
3. Energy 限制候选数量。
4. accepted candidate 进入持久 Queue。
5. 多 Worker 独立容器执行。
6. Coverage 影响下一轮反馈和选种。
7. 高价值候选进入 Corpus。
8. 符合条件的候选晋升并再次被选择。
9. Budget/Stagnation 正常结束 Campaign。

### 24.2 并发与恢复

1. 内存 Queue 有界。
2. SQLite 业务提交单写者。
3. Lease 可续约、过期、回收。
4. Worker 可以乱序完成，结果按 dispatch sequence 缓冲和提交且不重复。
5. 重启可恢复。
6. PAUSED 时无活动 Worker/容器。
7. Resume 拒绝 digest 漂移。
8. 单任务错误不退出 Campaign。
9. 系统错误安全暂停。
10. 清理不误删其他 Campaign。

### 24.3 可追溯性

```text
Campaign Manifest
  -> EnergyDecision
  -> Parent Seed
  -> Mutation Batch/Candidate/Operator
  -> WorkItem/Attempt/Execution
  -> Trajectory/Replay
  -> CoverageResult/Risk Evidence
  -> Corpus Reasons
```

引用 digest 不匹配必须显式失败。

### 24.4 本机完成标准

- 单元、集成、Docker 多轮和恢复 E2E 通过。
- 本机基础设施 Soak 达到约定时长。
- 无不可恢复崩溃。
- 无遗留 Campaign 容器和 volume。
- 可暂停、退出、恢复和导出。
- Fake/RuleBased 不被描述为真实语义质量。

### 24.5 Server-gated 标准

- 真实模型和镜像 digest 锁定。
- 受控模型网络在 Linux 验收。
- 1 至 2 个 Target Profile smoke 通过。
- 24 小时有效 Fuzzing。
- 覆盖和停滞原因可查询。
- GPU/CPU/内存/模型队列无不可接受泄漏或饥饿。
- Mutator 有效率、重复率和吞吐有记录。
- 恢复演练成功。

服务器条件未满足不阻塞代码合并，但 M5“真实模型持续探索”保持待验收。

## 25. 风险与策略

| 风险 | 影响 | 处理 |
|---|---|---|
| 多 Store 无跨库事务 | 半提交 | 固定顺序、幂等 API、状态机和 Recovery |
| 随机 execution ID | 无法判断已执行 | 按 work/attempt 派生 ID |
| SQLite 并发写 | locked/线程错误 | Coordinator 单写者 |
| 无限生成 | 磁盘/内存增长 | bounded queue、max pending、按容量生成 |
| 高能量 Seed 饥饿其他 Seed | 探索坍缩 | Weighted Virtual Runtime |
| RuleBased 空间耗尽 | 本机提前停滞 | 正确 STAGNATED，Soak Probe 单独统计 |
| Probe 污染覆盖 | 虚假增长 | measurement class 隔离 |
| 取消中断 cleanup | 遗留容器 | shield、cleanup timeout、恢复对账 |
| 全局 orphan cleanup | 误删其他 Campaign | Campaign label + active allowlist |
| retry 风暴 | 服务雪崩 | 有限 retry、退避、断路器 |
| 基础设施错误惩罚 Seed | 错误淘汰 | 只用正常 Coverage 更新 stagnation |
| Corpus/SeedPool 混淆 | 种子爆炸 | 独立模型与晋升 |
| 时间进入公式 | 恢复漂移 | logical iteration + virtual runtime |
| 模型配置漂移 | 不可比较 | Manifest 锁 digest |
| Agent/Mutator 抢 GPU | OOM/延迟 | server-gated 容量与独立配额 |
| 24h 只有重复输入 | 探索无意义 | Soak 与语义 Campaign 分开 |
| 风险不可达 | 浪费预算 | CampaignRiskScope 和停滞报告 |
| Prompt 日志泄漏 | 内容扩散 | 默认摘要，显式导出 |
